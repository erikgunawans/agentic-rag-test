# Module 4: Metadata Extraction

⚠️ **Medium** — Multiple touchpoints (DB, new service, ingestion pipeline, retrieval RPC, chat prompt, frontend), but each change is well-scoped and follows established patterns.

## Context

Modules 1-3 built a working RAG pipeline: upload documents, chunk + embed, retrieve via pgvector, and chat with context. But retrieved chunks lack document-level context — the LLM gets raw text snippets with no information about what document they came from, who wrote it, or what it's about. Module 4 adds LLM-powered structured metadata extraction during ingestion, stores it alongside documents, surfaces it in the UI, and enriches retrieval so the chat LLM gets document context with each chunk.

## Design Decisions

1. **JSONB column on `documents` table** (not a separate table) — metadata is 1:1 with documents; JSONB keeps schema simple while supporting GIN-indexed queries
2. **Extract during ingestion** (after parse, before chunk) — full parsed text is already in memory; ensures metadata is always available for completed documents
3. **Use user's selected LLM model** via OpenRouter — respects their choice, no extra config needed
4. **Fixed Pydantic schema** (title, author, date_period, category, tags, summary) — PRD says "no admin UI"; fixed schema covers the learning goals without over-engineering
5. **Enrich retrieval context** — return `document_id` + metadata from matched chunks via a new RPC function; include document context (filename, category, tags) alongside each chunk in the system prompt

## Tasks

### 1. Database Migration
- **File**: `supabase/migrations/005_document_metadata.sql`
- Add `metadata jsonb` column to `documents` (nullable for backward compat)
- Create GIN index on `metadata` for JSONB containment queries
- Create new RPC `match_document_chunks_with_metadata` that joins `document_chunks` with `documents` to return chunk content + document filename + document metadata + similarity score; accepts optional `filter_category text` parameter to filter by `metadata->>'category'`
- Keep the old `match_document_chunks` RPC intact (backward compat)
- **Validation**: Apply in Supabase dashboard; verify column, index, and RPC exist; existing docs have `metadata = NULL`

### 2. Pydantic Metadata Model
- **File**: `backend/app/models/__init__.py` (new, empty)
- **File**: `backend/app/models/metadata.py` (new)
- Define `DocumentMetadata(BaseModel)`:
  - `title: str` — document title or best guess
  - `author: str | None = None` — author if identifiable
  - `date_period: str | None = None` — date/period referenced (e.g. "2024", "Q3 2023")
  - `category: str` — one of: technical, legal, business, academic, personal, other
  - `tags: list[str]` — 3-7 descriptive keywords
  - `summary: str` — 2-3 sentence summary
- **Validation**: Import in Python shell, verify serialization/deserialization

### 3. Metadata Extraction Service
- **File**: `backend/app/services/metadata_service.py` (new)
- Class `MetadataService` with `AsyncOpenAI` client pointing to OpenRouter (same pattern as `openrouter_service.py`)
- Method `extract_metadata(text: str, model: str | None = None) -> DocumentMetadata | None`:
  - Truncate to first ~4000 tokens via tiktoken `cl100k_base` (metadata concentrates in title pages/intros)
  - System prompt instructs extraction of the fixed fields as JSON
  - Call `chat.completions.create()` with `response_format={"type": "json_object"}` (widely supported by OpenRouter models)
  - Parse response with `DocumentMetadata.model_validate_json()`
  - If parsing fails, log warning and return `None` (metadata is enrichment, not critical path)
  - Decorate with `@traceable` for LangSmith
- **Validation**: Manual test with sample text; check LangSmith trace

### 4. Integrate Extraction into Ingestion Pipeline
- **File**: `backend/app/services/ingestion_service.py`
  - Import `MetadataService`, instantiate at module level
  - Add `llm_model: str | None = None` parameter to `process_document()`
  - After `text = parse_text(...)`, before chunking: call `metadata_service.extract_metadata(text, model=llm_model)`
  - Wrap extraction in try/except — on failure, log and set `metadata_dict = None`
  - Add `"metadata": metadata_dict` to the final `documents.update()` call (alongside `status`, `chunk_count`)
- **File**: `backend/app/routers/documents.py`
  - Pass `llm_model=user_settings["llm_model"]` to `process_document` background task
  - Add `metadata` to `list_documents()` select fields
  - Add `GET /documents/{doc_id}/metadata` endpoint (validates ownership, returns metadata or null)
- **Validation**: Upload a document; verify ingestion completes, `metadata` JSONB populated in DB, `GET /documents` returns metadata field

### 5. Metadata-Enhanced Retrieval
- **File**: `backend/app/services/embedding_service.py`
  - Add method `retrieve_chunks_with_metadata(query, user_id, top_k, threshold, model, category=None) -> list[dict]`
  - Calls new `match_document_chunks_with_metadata` RPC
  - Returns list of dicts: `{content, doc_filename, doc_metadata, similarity}`
  - Decorated with `@traceable`
- **File**: `backend/app/routers/chat.py`
  - Replace `retrieve_chunks()` with `retrieve_chunks_with_metadata()`
  - Update system prompt construction — for each chunk, include source context:
    ```
    [Source: "filename.pdf" | Category: technical | Tags: python, rag, embeddings]
    {chunk content}
    ```
  - This gives the LLM document-level context for each retrieved chunk
- **Validation**: Upload a doc, chat with a relevant question, verify response leverages document context; check LangSmith trace shows enhanced retrieval

### 6. Frontend Types

### 7. Frontend — Document List with Metadata
