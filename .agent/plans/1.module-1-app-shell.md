# Module 1 Plan: App Shell + Observability

## Context
Module 1 builds the foundation of the RAG application: authentication, a chat interface with threaded conversations, OpenAI Responses API integration (managed RAG — OpenAI handles threads and file_search internally), and LangSmith observability. This is the "black box" RAG phase — Module 2 will replace the Responses API with custom retrieval (Chat Completions + pgvector).

No code exists yet. This plan scaffolds the entire project from scratch.

## Key Architecture Decisions

### DB Schema (forward-compatible for Module 2+)
- `threads` — maps user → OpenAI thread ID (+ title, timestamps)
- `messages` — stores all messages for display (Module 2 repurposes this for stateless history)
- pgvector extension enabled now (used from Module 2)
- No `documents`/`chunks`/`embeddings` tables yet — those are Module 2

### OpenAI Responses API pattern
- `client.responses.create()` — NOT `chat.completions.create()`
- Thread continuity via `previous_response_id` chaining (stored per thread in DB)
- `file_search` tool attached if `OPENAI_VECTOR_STORE_ID` is set (empty store in Module 1 — populated in later modules or manually via OpenAI dashboard)
- Streaming via `client.responses.stream()` context manager → SSE to frontend
- Model: `gpt-4o-mini`

### Auth pattern
- Frontend: `@supabase/supabase-js` for auth, JWT stored in localStorage
- Backend: FastAPI `HTTPBearer` dependency → `client.auth.get_user(token)` validates JWT
- RLS: all DB queries automatically scoped to `auth.uid()`

### Streaming
- Backend: FastAPI `StreamingResponse` with `text/event-stream` content type
- Frontend: `EventSource` or `fetch` with `ReadableStream` to consume SSE

## Constraints
- Backend: Python venv at `backend/venv/` (non-negotiable)
- No LangChain/LangGraph — raw OpenAI SDK + `@traceable` from langsmith only
- RLS on all tables — users see only their own data
- SSE for streaming (not WebSockets)
- Config via env vars — no admin UI
- Pydantic for structured outputs
