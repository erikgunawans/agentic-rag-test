# PJAA CLM Platform: Gap Analysis & Feature Specifications

**Complexity**: đź”´ **Complex** - 14 features across 3 phases, break into sub-plans before executing

## Context

PJAA (PT Pembangunan Jaya Ancol Tbk) completed a 53-question stakeholder survey identifying 7 critical findings for a CLM & Regulatory Compliance platform. This spec maps the **gaps between our current system and the survey requirements**, then specifies the features needed to close each gap.

**Source**: `References/PJAA-Research-Synthesis-CLM-Compliance.docx.md`

**Current system**: A 10-module RAG application with chat, document ingestion (7 formats), hybrid search, tool calling, multi-agent orchestration, document tools (creation, comparison, compliance check, contract analysis), RBAC, conversation branching, and bilingual i18n (ID/EN).

**What the survey demands**: Regulatory intelligence, obligation lifecycle management, enhanced drafting, mobile executive approval, WhatsApp notifications, MFA/security hardening, AI confidence scoring with HITL gates, audit trails, executive dashboards, DMS integration, and UU PDP compliance.

## Phase 1: Weeks 1-8 (Go-Live Foundation)

### Feature 1: Audit Trail & Activity Logging
**Priority**: Critical | **Survey**: F1, F2, F6 | **Weeks 1-2**

**Problem**: Zero audit logging. PJAA had compliance findings within 12 months and manages audit findings in spreadsheets.

**Solution**: Universal `audit_logs` table + FastAPI middleware for automatic capture + admin query UI.

**Database** (migration `010_audit_trail.sql`):
- `audit_logs` table: `id`, `user_id`, `user_email`, `action`, `resource_type`, `resource_id`, `details` (jsonb), `ip_address`, `created_at`
- Indexes on `user_id`, `created_at`, `action`, `resource_type`
- No RLS (admin-only access via service role)

**Backend**:
- New `backend/app/services/audit_service.py` - `log_action()` function
- New `backend/app/routers/audit.py` - `GET /admin/audit-logs` (paginated + filtered), `GET /admin/audit-logs/export` (CSV)
- FastAPI middleware logging all mutating requests (POST/PATCH/PUT/DELETE)

**Frontend**:
- New `AuditLogPage.tsx` under AdminGuard at `/admin/audit`
- Date range, user, action type filters + CSV export button

**Validation**: Create a document, run compliance check, verify both produce audit log entries. Admin can filter and export CSV.

### Feature 2: AI Confidence Scoring & HITL Gates
**Priority**: Critical | **Survey**: F7 | **Weeks 2-4**

**Problem**: No confidence indicators on LLM outputs. PJAA's top concern is hallucination; they require >85% confidence threshold before automated decisions.

**Solution**: Add `confidence_score` to all document tool responses. Results below threshold enter a review queue requiring human approval.

**Database** (migration `011_confidence_hitl.sql`):
- ALTER `document_tool_results`: add `confidence_score` (float), `review_status` (auto_approved/pending_review/approved/rejected), `reviewed_by`, `reviewed_at`, `review_notes`
- ALTER `system_settings`: add `ai_confidence_threshold` (float, default 0.85)

**Backend**:
- Modify `backend/app/services/document_tool_service.py` `_llm_json()` to request `confidence_score` in every structured response
- New endpoints: `GET /document-tools/review-queue`, `PATCH /document-tools/review/{id}`

**Frontend**:
- Confidence badge component (green >= 0.85, yellow 0.60-0.84, red < 0.60)
- Review queue page at `/review`
- Warning banner on pending_review results
- Confidence threshold slider in admin settings

**Validation**: Run compliance check, verify confidence score appears. Set threshold above result to confirm pending_review. Approve from review queue.

### Feature 3: Enhanced Drafting Workbench
**Priority**: High | **Survey**: F3 | **Weeks 3-5**

**Problem**: Only 4 doc types (generic, NDA, sales, service), no clause library, no template management, no per-clause risk scoring. PJAA needs 7 types and consistent templates.

**Solution**: Clause library (searchable, reusable clauses with risk scores), template management, 7 doc types, per-clause risk scoring in generated documents.

**Database** (migration `012_drafting_workbench.sql`):
- `clause_library` table: `id`, `user_id`, `title`, `content`, `category`, `applicable_doc_types[]`, `risk_level`, `language`, `tags[]`, `is_global`
- `document_templates` table: `id`, `user_id`, `name`, `doc_type`, `content`, `fields_schema` (jsonb), `is_global`
- RLS: users see own + global clauses/templates

**Backend**:
- New `backend/app/routers/clause_library.py` - CRUD + search
- New `backend/app/routers/templates.py` - CRUD
- Extend `POST /document-tools/create` to accept `clause_ids[]` and `template_id`

**Frontend**:
- Expand doc type dropdown: add `vendor`, `jv`, `property_lease`, `employment`, `sop_resolution`
- New `ClauseLibraryPage.tsx` at `/clause-library`
- Clause selector panel during document creation
- Template selector + "Save as Template" button
- Per-clause risk badges in generated document view

**Validation**: Create document using clause library selections and template. Verify all 7 doc types work. Verify per-clause risk badges.

### Feature 4: Obligation Lifecycle Tracker
**Priority**: Critical | **Survey**: F2 | **Weeks 4-6**

**Problem**: 201-500 active contracts with no centralized obligation register. Contract analysis extracts obligations but stores them only as JSONB blobs in `document_tool_results`. No deadline tracking or notifications.

**Solution**: Structured `obligations` table with deadline tracking, status management, and reminder system.

**Database** (migration `013_obligation_tracker.sql`):
- `obligations` table: `id`, `user_id`, `document_id`, `analysis_id`, `party`, `obligation_text`, `obligation_type` (payment/reporting/renewal/termination/compliance/general), `deadline`, `recurrence`, `status` (active/completed/overdue/cancelled/upcoming), `priority`, `reminder_days`, `contract_title`
- RLS: users manage own obligations
- `check_overdue_obligations()` PL/pgSQL function for status updates

**Backend**:
- New `backend/app/routers/obligations.py` - CRUD + filters + summary stats
- New `backend/app/services/obligation_service.py`
- `POST /obligations/extract/{analysis_id}` - extract from existing analysis into structured rows
- `POST /obligations/check-deadlines` - trigger overdue/upcoming status update
- `GET /obligations/summary` - counts by status for dashboard

**Frontend**:
- New `ObligationsPage.tsx` at `/obligations` with table/kanban view
- Deadline calendar component (week/month view)
- "Import Obligations" button after contract analysis
- Badge counts on nav for overdue obligations

**Validation**: Run contract analysis, click "Import Obligations", verify obligations appear in tracker. Set deadline to yesterday, run check-deadlines, verify overdue status.

### Feature 7: Bahasa Indonesia Full-Text Search
**Priority**: High | **Survey**: F7 | **Weeks 1-2**

**Problem**: FTS trigger hardcodes `to_tsvector('english', content)`. Indonesian documents won't be properly searchable.

**Solution**: Switch to `'simple'` text search config (language-agnostic, works for both EN and ID).

**Database** (migration `016_bahasa_fts.sql`):
- Replace FTS trigger: `to_tsvector('simple', content)` instead of `'english'`
- Update `match_document_chunks_fulltext` RPC: `websearch_to_tsquery('simple', ...)`
- Backfill existing rows

**Key file**: `supabase/migrations/006_hybrid_search.sql` (the trigger and RPC being replaced)

**No backend/frontend changes needed** - existing search automatically benefits.

**Validation**: Upload Indonesian document, search for Indonesian terms (e.g., "perjanjian", "kewajiban"), verify results returned.

## Phase 2: Weeks 9-16 (Enterprise Capabilities)

### Feature 8: Regulatory Intelligence Engine
**Priority**: Critical | **Survey**: F1 | **Weeks 9-12**

**Problem**: Zero internal regulatory change tracking. Entirely reliant on external consultants. Had compliance sanctions within 12 months.

**Solution**: Automated regulatory crawler for JDIH, IDX, Perda DKI + LLM-powered relevance scoring against user's contract corpus.

**Database** (migration `017_regulatory_intelligence.sql`):
- `regulatory_sources` table: crawl targets (URL, type, schedule)
- `regulatory_updates` table: crawled regulation content + embedding + relevance scores
- `regulatory_alerts` table: per-user alerts based on relevance scoring

**Backend**:
- New `backend/app/services/regulatory_crawler_service.py` - httpx + BeautifulSoup crawlers
- New `backend/app/services/regulatory_relevance_service.py` - embedding similarity scoring
- New `backend/app/routers/regulatory.py` - source management, update feed, alert inbox

**Frontend**:
- New `RegulatoryPage.tsx` at `/regulatory` with update feed + source management (admin) + alert inbox
- Alert badge on nav icon

**Risk**: JDIH/IDX crawling may be fragile. Use configurable CSS selectors stored in `regulatory_sources.metadata`.

**Validation**: Configure JDIH source, trigger crawl, verify updates stored. Check relevance scoring against uploaded contracts.

### Risks & Mitigations
1. **WhatsApp Business API approval** takes 1-3 weeks. Start Meta Business verification early.
2. **Dokmee REST API docs** must be obtained from PJAA IT. Spike API exploration in Phase 1.
3. **JDIH/IDX crawling** may be fragile. Use configurable CSS selectors in `regulatory_sources.metadata`.
4. **Data residency** (asia-southeast2): Verify Supabase region and OpenRouter routing.
5. **No LangChain/LangGraph**: All LLM calls via raw OpenAI SDK through `OpenRouterService`. Use existing `_llm_json` pattern.
