```typescript
interface ToolCallListProps {
  toolCalls: ToolCallRecord[]
  sandboxStreams?: Map<string, { stdout: string[]; stderr: string[] }>
}

export function ToolCallList({ toolCalls, sandboxStreams }: ToolCallListProps) {
  // D-P11-01 + D-P11-05: split execute_code calls (with tool_call_id) into a
  // separate vertical-stack list rendered with `gap-6`. Other tools keep the
  // existing `space-y-0.5` rhythm.
  const sandboxCalls: ToolCallRecord[] = []
  const otherCalls: ToolCallRecord[] = []
  for (const tc of toolCalls) {
    if (tc.tool === 'execute_code' && tc.tool_call_id) {
      sandboxCalls.push(tc)
    } else {
      otherCalls.push(tc)
    }
  }

  return (
    <>
      {sandboxCalls.length > 0 && (
        <div className="flex flex-col gap-6 mb-2">
          {sandboxCalls.map((tc) => {
            const tcid = tc.tool_call_id as string
            const live = sandboxStreams?.get(tcid)
            const out =
              typeof tc.output === 'object' && tc.output !== null
                ? (tc.output as Record<string, unknown>)
                : {}
            const persistedStdout = (out.stdout_lines as string[] | undefined)
              ?? (typeof out.stdout === 'string' ? (out.stdout as string).split('\n') : [])
            const persistedStderr = (out.stderr_lines as string[] | undefined)
              ?? (typeof out.stderr === 'string' ? (out.stderr as string).split('\n') : [])
            const stdoutLines = live?.stdout ?? persistedStdout ?? []
            const stderrLines = live?.stderr ?? persistedStderr ?? []
            const persistedStatus =
              (tc.status as 'success' | 'error' | 'timeout' | null | undefined) ?? null
            const status: 'pending' | 'running' | 'success' | 'error' | 'timeout' =
              live ? 'running' : (persistedStatus ?? 'success')
            const code = (tc.input as { code?: string })?.code ?? ''
            const files =
              (out.files as { filename: string; size_bytes: number; signed_url: string }[] | undefined)
              ?? []
            const executionId = (out.execution_id as string | undefined)
              ?? (out.id as string | undefined)
            const executionMs = out.execution_ms as number | undefined
            const errorType = out.error_type as string | undefined
            return (
              <CodeExecutionPanel
                key={tcid}
                toolCallId={tcid}
                executionId={executionId}
                code={code}
                status={status}
                executionMs={executionMs}
                stdoutLines={stdoutLines}
                stderrLines={stderrLines}
                files={files}
                errorType={errorType}
              />
            )
          })}
        </div>
      )}
      {otherCalls.length > 0 && (
        <div className="space-y-0.5">
          {otherCalls.map((tc, i) => (
            <ToolCallCard
              key={i}
              tool={tc.tool}
              input={tc.input}
              output={tc.output as Record<string, unknown>}
            />
          ))}
        </div>
      )}
    </>
  )
}
```

The output-field name fallback chain (`stdout_lines` → split `stdout` on newline) is defensive: Phase 10's persisted record may store stdout as either a string or an array depending on which branch wrote it. The executor MUST verify which key Phase 10 actually stores by reading one sample row from `code_executions` (or by reading `backend/app/services/sandbox_service.py`'s `_run_in_session` return shape) and tightening the read accordingly. If Phase 10 always stores `stdout` as a single string, the fallback is sufficient.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Persisted tool record → CodeExecutionPanel props | `tc.input.code`, `tc.output.stdout`, `tc.output.stderr`, `tc.output.files` all originate from cloud LLM tool output and the sandbox; rendered by Plan 11-06's React-auto-escaped components. |
| Live SSE Map → CodeExecutionPanel | `sandboxStreams.get(tool_call_id)` pulls already-anonymized lines (anonymization happens in chat.py per Phase 10-05). |
| MessageView → ToolCallList prop | `sandboxStreams` is a per-render reference to the hook's Map; React handles identity. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-07-1 | Information Disclosure | sandboxStreams entries from prior message bleed into current panel | mitigate | Plan 11-05's 3-site reset (handleSelectThread, sendMessageToThread, finally) ensures the Map is empty between turns. The `tool_call_id` keys are UUIDs — collisions across panels are statistically impossible. |
| T-11-07-2 | Tampering | Forged `tool_call_id` in persisted record causes mis-routing | accept | A user could in principle write a forged `tool_call_id` by directly editing their own messages JSONB row, but RLS prevents cross-user writes and the worst case is rendering an empty CodeExecutionPanel. No cross-user impact. |
| T-11-07-3 | XSS | Output rendering | mitigate | All rendering goes through Plan 11-06's React-auto-escaped JSX. This plan only adds the routing switch — no new rendering paths. |
| T-11-07-4 | Spoofing | Legacy execute_code call rendered as a sandbox panel | mitigate | Switch checks BOTH `tc.tool === 'execute_code'` AND `tc.tool_call_id` truthy. Legacy rows lacking `tool_call_id` route to the generic `ToolCallCard` via the new `TOOL_CONFIG.execute_code` entry. |

ASVS L1 surface: zero new auth surface; relies on existing JWT-bearing `apiFetch` (used inside Plan 11-06's panel, not this routing layer). No new credential.
