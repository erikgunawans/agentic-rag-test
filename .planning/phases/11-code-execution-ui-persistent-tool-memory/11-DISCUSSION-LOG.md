## Round 1 — Core architecture

### Q1: Where does the Code Execution Panel render relative to the existing ToolCallList in MessageView.tsx?

**Options presented:**
- Replace ToolCallCard for execute_code only *(Recommended)*
- Render CodeExecutionPanel as a separate block above the assistant bubble
- Inline inside the assistant bubble

**User choice:** Replace ToolCallCard for execute_code only

**Why this matters:** Determines whether `ToolCallList.tsx` gets a per-tool switch (smallest blast radius) or a parallel render path is added (risk of duplicate display). Locked into D-P11-01.

---

### Q2: During and after streaming, where does the Code Execution Panel read its stdout/stderr/files data from?

**Options presented:**
- Live SSE during stream → persisted message after refetch *(Recommended)*
- Always read from persisted message
- Always read from live SSE buffer (keep buffer indefinitely)

**User choice:** Live SSE during stream → persisted message after refetch

**Why this matters:** Defines the state-management contract between `useChatState` and `CodeExecutionPanel`. The chosen path mirrors the existing `streamingContent` → refetch pattern that's been battle-tested for assistant text. Locked into D-P11-02.

---

### Q3: How does the new history-load reconstruction handle pre-Phase-11 messages.tool_calls rows that lack tool_call_id?

**Options presented:**
- Skip reconstruction for legacy rows; include only role/content *(Recommended)*
- Synthesize tool_call_ids on-the-fly during reconstruction
- One-shot backfill migration to add tool_call_id to existing rows

**User choice:** Skip reconstruction for legacy rows; include only role/content

**Why this matters:** Production data integrity vs. backwards-feature-coverage trade-off. Chosen path honors PRD's "no schema migration" guidance. Locked into D-P11-03.

---

### Q4: How large can a single tool result be in messages.tool_calls.calls[N].output, and what happens at the boundary?

**Options presented:**
- 50 KB per call, head-truncate with marker *(Recommended)*
- 100 KB per call, head+tail with middle elision
- Per-tool cap (search_documents 30 KB, execute_code 100 KB, others 50 KB)

**User choice:** 50 KB per call, head-truncate with marker

**Why this matters:** Caps growth on the messages JSONB column, sets an upper bound on per-turn token contribution from prior tool results, and keeps the truncation logic simple (one constant, one path). Locked into D-P11-04.

## Round 2 — UI behavior & MEM scope

### Q5: When the LLM makes multiple execute_code calls in one assistant turn, how should the panels render?

**Options presented:**
- One independent panel per call, stacked vertically *(Recommended)*
- Merged terminal across calls in the same turn

**User choice:** One independent panel per call, stacked vertically

**Why this matters:** Mirrors `ToolCallList`'s one-card-per-call existing semantics; preserves per-call file-card grouping. Locked into D-P11-05.

---

### Q6: How do users interact with files generated by execute_code?

**Options presented:**
- Download-only via signed URL *(Recommended)*
- Click-to-preview text files in slide-in drawer + Download for binary

**User choice:** Download-only via signed URL

**Why this matters:** Sandbox outputs are mostly binary (matplotlib, PowerPoint, Excel) — the preview path would rarely fire. Saves frontend complexity. Locked into D-P11-06.

---

### Q7: Does persistent tool memory (MEM-01..03) apply to all tools or only execute_code?

**Options presented:**
- All tools *(Recommended)*
- Only execute_code; other tools keep current behavior

**User choice:** All tools — search_documents, query_database, web_search, load_skill, read_skill_file, kb_*, execute_code

**Why this matters:** PRD §Feature 5 examples (UUIDs, search results, file listings) come from non-sandbox tools. Single uniform path; one code change. Locked into D-P11-07.

---

### Q8: How should ToolCallRecord (backend/app/models/tools.py) be extended for MEM-01?

**Options presented:**
- Add tool_call_id + status + capped result *(Recommended)*
- Minimal — add only tool_call_id; derive status from output/error at render time
- Full PRD shape — tool_call_id, status, summary, result (replace 'output' field)

**User choice:** Add tool_call_id + status + capped result

**Why this matters:** Frontend ToolCallCard.tsx and any other readers continue to work without changes (existing `output` field preserved). Backwards-compatible additive change. Locked into D-P11-08.

## Round 3 — Polish & enforcement

### Q9: What is the default state of the collapsible code preview block in CodeExecutionPanel?

**Options presented:**
- Collapsed by default; user clicks chevron to expand *(Recommended)*
- Expanded during streaming, auto-collapse after success
- Always expanded

**User choice:** Collapsed by default; user clicks chevron to expand

**Why this matters:** Reduces visual noise especially when multiple panels stack. Matches PRD's "Collapsible monospace block" wording (collapsed-first implication). Locked into D-P11-09.

---

### Q10: When a chat session has PII redaction ON and history reconstruction adds a prior tool result message, how is that historical content protected?

**Options presented:**
- Anonymize reconstructed tool messages through the existing per-history anonymizer *(Recommended)*
- Skip reconstruction entirely when redaction is ON
- Reconstruct but skeleton-emit (drop the result body, keep metadata)

**User choice:** Anonymize reconstructed tool messages through the existing per-history anonymizer

**Why this matters:** Honors the v1.0 D-89 privacy invariant (no real PII in cloud-LLM payloads) without disabling MEM-03 for the platform's main user base. Single code path; reuses the battle-tested anonymizer. Locked into D-P11-10.

---

### Q11: Where in the backend is the 50 KB head-truncate enforced?

**Options presented:**
- Inside ToolCallRecord at construction *(Recommended)*
- In chat.py at persist time only
- In a shared utility module imported by chat.py

**User choice:** Inside ToolCallRecord at construction

**Why this matters:** Single source of truth in `models/tools.py`; every code path that builds a record gets truncation automatically. Status enum lives next to the validator for symmetry. Locked into D-P11-11.

## Deferred Ideas

- **Per-tool size caps** — uniform 50 KB chosen; revisit only on prod bloat observation.
- **Backfill migration** — discarded; legacy rows skip reconstruction.
- **Click-to-preview text files** — discarded; sandbox outputs are mostly binary.
- **Merged terminal across calls** — discarded in favor of per-call panels.
- **Always-expanded code preview** — discarded; collapsed-by-default chosen.
- **Renaming `output` → `result`** — discarded for backwards compatibility.
- **`summary` field on ToolCallRecord** — not added; output covers data, status covers state.
