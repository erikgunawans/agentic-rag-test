<objective>
Make the PII domain-term deny list at `backend/app/services/redaction/detection.py`
runtime-configurable via a new `system_settings.pii_domain_deny_list_extra` column,
while preserving the existing baked-in `_DENY_LIST_CASEFOLD` frozenset baseline
(zero-regression on production thread `bf1b7325`). Cache the union'd frozenset
locally for 60s (matching `get_system_settings()` TTL); never read settings inside
the chat hot path (`detect_entities`).
</objective>

    <title>Add cached frozenset builder to detection.py</title>
    <type>execute</type>
    <action>
      In `backend/app/services/redaction/detection.py`, just below the
      `_DENY_LIST_CASEFOLD: frozenset[str] = frozenset({...})` block (currently
      lines 64-86), add a new module-private cache + builder. Concrete code:

      ```python
      # REDACT-01 / D-P16-03 / D-P16-04: 60s-cached union of the baked-in deny
      # list and the runtime extras column from system_settings. The cache
      # holds a frozenset (preserves O(1) lookup in _is_domain_term). Rebuild
      # is keyed on the runtime-extras string so that updates within the TTL
      # are picked up at most once per 60s — matching get_system_settings().
      _DENY_LIST_TTL_SECONDS: float = 60.0
      _deny_list_cache: frozenset[str] = _DENY_LIST_CASEFOLD
      _deny_list_cache_ts: float = 0.0
      _deny_list_cache_extras_key: str = ""


      def _parse_extras_csv(raw: str) -> frozenset[str]:
          """Parse comma-separated extras into a casefolded frozenset.

          Whitespace around items is stripped; empty items are dropped.
          Empty / None input yields an empty frozenset. Mirrors the parsing
          rule used elsewhere in detection.py for pii_*_entities columns,
          but case-folds (not upper) because _is_domain_term casefolds.
          """
          if not raw:
              return frozenset()
          return frozenset(
              item.strip().casefold()
              for item in raw.split(",")
              if item.strip()
          )


      def _get_active_deny_list() -> frozenset[str]:
          """Return the currently-active deny list as a frozenset.

          Cached for 60s. The cache key includes the raw extras string so a
          deployer updating system_settings.pii_domain_deny_list_extra is
          honored on the next call after the 60s TTL expires (D-P16-03).
          On any error reading system_settings, falls back to the baked-in
          frozenset (zero-regression invariant from D-P16-02).
          """
          global _deny_list_cache, _deny_list_cache_ts, _deny_list_cache_extras_key
          now = time.time()
          if (now - _deny_list_cache_ts) < _DENY_LIST_TTL_SECONDS:
              return _deny_list_cache

          try:
              from app.services.system_settings_service import get_system_settings
              row = get_system_settings() or {}
              extras_raw = row.get("pii_domain_deny_list_extra") or ""
          except Exception as exc:  # pragma: no cover - defensive
              logger.warning("deny_list extras read failed; using baseline only: %s", exc)
              extras_raw = ""

          if extras_raw == _deny_list_cache_extras_key and _deny_list_cache_ts > 0.0:
              # Same extras string — refresh ts only, reuse the frozenset.
              _deny_list_cache_ts = now
              return _deny_list_cache

          extras_set = _parse_extras_csv(extras_raw)
          _deny_list_cache = frozenset(_DENY_LIST_CASEFOLD | extras_set)
          _deny_list_cache_ts = now
          _deny_list_cache_extras_key = extras_raw
          return _deny_list_cache
      ```

<threat_model>
**Surface:** PII detection layer reading from `system_settings`.

**Threats:**
- **T1 (Low):** A malicious admin sets `pii_domain_deny_list_extra` to a huge string (millions of chars) → memory blowup. *Mitigation:* TEXT column accepts up to ~1GB but Postgres row limit (TOAST) and the 60s cache means the parsed frozenset lives in process memory once; for sane admin use (<10KB list), this is non-issue. If we need a hard cap, add `CHECK (length(pii_domain_deny_list_extra) <= 65536)` in a follow-up migration. Out of scope here.
- **T2 (Low):** A malicious admin uses the deny list to *exclude* real PII patterns (e.g. add common Indonesian first names) → real PII leaks past the redaction layer. *Mitigation:* admin role required to write `system_settings`; audit logging on `update_system_settings()` already exists; egress filter at `redaction/egress.py` is unaffected by deny list (filters BEFORE registry insertion is upstream of egress check). The deny list does not bypass the egress filter — that's the canonical-only safety net (D-89).
- **T3 (Low):** Cache poisoning between requests. *Mitigation:* `_deny_list_cache` is module-global within a single Python process; multi-process deployments (gunicorn workers) each maintain their own cache. The 60s TTL bounds inconsistency. Consistent with how `pii_*_entities` settings already behave.

**Block on:** No HIGH severity threats. All findings LOW; no blockers per `security_block_on=high` config.
</threat_model>
