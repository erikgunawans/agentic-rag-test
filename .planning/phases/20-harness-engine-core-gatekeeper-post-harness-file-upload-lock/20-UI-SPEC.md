# Phase 20 — UI Design Contract

> Visual and interaction contract for the three new frontend surfaces shipped by Phase 20: **Locked Plan Panel variant**, **HarnessBanner**, and **FileUploadButton**. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase has **no greenfield design system work** — every token below is already declared in `frontend/src/index.css` and consumed by the existing 2026 Calibrated Restraint surfaces. Phase 20 only inherits and extends.

## Spacing Scale

Tailwind defaults (4px multiples). Values declared by usage in this phase:

| Token | Value | Usage in Phase 20 |
|-------|-------|-------------------|
| xs | 4px (`gap-1`, `p-1`) | Inline icon gap inside chips, tight badge padding |
| sm | 8px (`gap-2`, `p-2`, `px-2 py-1`) | Banner element gap, paperclip button hit area, tooltip body padding |
| md | 12–16px (`px-3 py-2`, `px-4 py-3`) | PlanPanel/HarnessBanner header padding (matches `WorkspacePanel` `px-4 py-3` exactly) |
| lg | 24px (`gap-6`) | Section breaks inside post-upload progress card |
| xl | 32px | Not used in Phase 20 |

**Hit-target exceptions:**
- Paperclip icon button: 32px square (`h-8 w-8`) — matches existing `InputActionBar` icons (Plus, FileText, Mic, Send). Not the 44pt iOS minimum, but consistent with the established input toolbar pattern.
- Cancel button on banner / panel header: 24px square (`h-6 w-6`) for the icon-only variant inside dense headers; 32px (`h-8 px-3`) when rendered as a labeled button.

**Locked panel header:** matches existing `PlanPanel` header (`px-4 py-3 border-b border-border/50`) byte-for-byte. Lock icon + harness-type label + Cancel slot is laid out with `gap-2` (8px) between icon and text, `gap-1` (4px) between text and Cancel.

60/30/10 split honored — Phase 20 introduces NO new color tokens. All swatches resolve from `frontend/src/index.css` CSS variables.

| Role | Value (light → dark) | Usage in Phase 20 |
|------|----------------------|-------------------|

| Destructive | Tailwind `red-600` / `red-50` family (matches `AgentStatusChip` error variant) | Upload error toast text, file-too-large inline error, Cancel-confirmation Dialog destructive button |
| Success | Tailwind `green-600` / `green-500` family (matches `PlanPanel` `CheckCircle2`) | Upload success flash on file row (200ms) |

**Accent (purple, `--primary`) reserved-for list — Phase 20 ONLY uses it in:**
1. **Lock icon** in locked PlanPanel header (`text-primary` on the `Lock` lucide glyph at 16px)
2. **Tooltip arrow + border accent** on the lock-icon tooltip (already inherited from `tooltip.tsx` which uses `border-primary/20`)
3. **Phase progress dot** in HarnessBanner (`h-1.5 w-1.5 rounded-full bg-primary animate-pulse` — matches the `working` state of `AgentStatusChip`)

Explicit non-uses of accent in Phase 20:
- Locked panel surface = `bg-background`, NOT a purple wash
- HarnessBanner surface = `bg-background` with `border-b border-border/50`, NOT a purple banner
- Cancel button = `outline` shadcn variant (zinc border, transparent fill), NOT `destructive` and NOT primary purple. Only the **confirmation Dialog**'s confirm button uses `destructive` (red).
- Paperclip icon = `text-muted-foreground` default, `text-foreground` on hover, NEVER purple
- File upload progress bar = zinc track, `bg-foreground/60` fill, NOT purple

**Dark mode parity:** All three new components MUST render correctly in `.dark` mode. Tokens listed above resolve correctly in both themes; no hardcoded hex values are permitted (match `PlanPanel.tsx` and `WorkspacePanel.tsx` discipline).

### Locked PlanPanel variant — visual contract

**Header layout** (left → right, all in `flex items-center gap-2`):
1. `Lock` icon (lucide, 16px, `text-primary` purple) — wrapped in `Tooltip` shim, `TooltipContent` shows tooltip copy (table below). Tooltip MAY use `backdrop-blur` per CLAUDE.md transient-overlay rule.
2. Harness-type label — `text-sm font-semibold text-foreground` — value comes from `harnessRun.harnessType` (e.g. `"Contract Review"`, `"Smoke Echo"`). Pre-translated via `t('harness.type.<harnessType>')`.
3. `flex-1` spacer
4. Cancel button — `<Button variant="outline" size="sm">` with `X` icon (14px) + `t('common.cancel')` label. On click → opens **confirmation Dialog**.
5. Existing collapse chevron stays put (preserves user's collapse preference).

**List items** (locked variant): identical to current `PlanPanel` rendering — same status icons (zinc `Circle` / purple `Loader2 animate-spin` / green `CheckCircle2`), same `space-y-2`, same line-through on completed. **No** add/edit/delete controls are added in this variant (none exist today either; the contract is "do not add them later either, while harness is active").

**`paused` state rendering:** the rendering switch on `harnessRun.status` MUST include a `paused` arm even though Phase 20 never enters that state. The `paused` arm renders identically to `running`. This prevents a runtime crash when Phase 21 lands `llm_human_input`. Testable via a unit case that passes `harnessRun.status='paused'` and asserts no throw.

**Defense-in-depth (D-12):** UI hides any future mutation affordances when locked. Today there are none, so visually the diff is minimal — only the header changes.

### HarnessBanner — visual contract

**Slot:** rendered by `AppLayout.tsx` directly above the chat scroll region (sibling to `AgentStatusChip`'s sticky position). Sits in the chat header band, NOT the global app header rail.

**Surface:** `bg-background border-b border-border/50 px-4 py-2 flex items-center gap-2`. Solid surface — **no blur, no gradient, no purple wash**. Visual restraint per CLAUDE.md.

**Content** (left → right):
1. `h-1.5 w-1.5 rounded-full bg-primary animate-pulse` — pulsing accent dot (only purple element)
2. Title text — `text-sm font-semibold` — copy below
3. `flex-1` spacer
4. Cancel button — `<Button variant="outline" size="sm">` with `X` icon + `t('common.cancel')`. Same Dialog confirmation flow as the locked PlanPanel header Cancel.

**Terminal-state behavior** (`cancelled` / `failed`):
- Pulsing dot replaced by static colored icon: `cancelled` → muted-foreground `XCircle`; `failed` → red `AlertCircle`
- Title text becomes terminal copy (table below)
- Cancel button hidden
- Auto-unmounts after 3000ms (matches `AgentStatusChip` timing convention)

**Aria:** `role="status"` `aria-live="polite"` so screen readers announce phase transitions. Same accessibility model as `AgentStatusChip`.

### FileUploadButton — visual contract

**Slot:** inserted into `InputActionBar` as a new icon button positioned **immediately to the right of the existing `Plus` button** (slot 2). Hidden when `WORKSPACE_ENABLED=false`.

**Idle visual:** `Paperclip` lucide icon, 16px, in a `h-8 w-8 rounded-lg flex items-center justify-center text-muted-foreground hover:text-foreground hover:bg-muted transition-colors` button. Byte-identical class string to the existing `Plus` and `FileText` toolbar buttons in `InputActionBar.tsx` lines 32–37 — visual consistency is the contract.

**Click behavior:** opens native file picker (`<input type="file" hidden accept=".docx,.pdf">`). On selection:
1. Client-side validation: file size ≤ 25 MB, MIME ∈ {DOCX, PDF}. On fail → inline error toast (copy table) and abort.
2. POST `/threads/{id}/files/upload` (multipart). Show progress.
3. Server validates magic bytes (DOCX `PK\x03\x04`, PDF `%PDF-`). On 4xx → toast with backend error code mapped to localized copy. On 5xx → generic error toast.
4. On 200 → flash success on the new `WorkspacePanel` row (green border 200ms). No separate toast — the file appearing in the panel IS the confirmation.

**Progress UI:** mirrors `WorkspacePanel` download UX — small inline progress indicator under the input toolbar (`Loader2 animate-spin` + `t('upload.uploading')` + filename + bytes-uploaded / total). Cancel button on the in-flight upload aborts the `fetch` (AbortController). Supports **one upload at a time** in v1.3 (sequential queue if user clicks paperclip again — keeps state simple).

**Aria:** `aria-label={t('chat.attachFile')}`. The hidden `<input type="file">` is keyboard-activatable via the visible button.

**Disabled state:** when an upload is already in flight → button has `disabled` attribute + `opacity-50 cursor-not-allowed`. NOT disabled when a harness is active — D-13 explicitly keeps it usable so the user can pre-load files for the next harness.

## Interaction Contracts

### Locked PlanPanel — Cancel flow
1. User clicks header Cancel → opens `<Dialog>` (shadcn). Dialog title + body + actions per copywriting table.
2. Confirm → POST `/threads/{id}/harness/cancel` (planner specifies exact path).
3. SSE `harness_phase_error` (with `code: cancelled`) followed by `harness_complete` with terminal status `cancelled`.
4. Banner transitions to terminal state (3000ms), then unmounts. PlanPanel returns to its pre-locked variant on next render (still shows the completed/in-progress phases as `agent_todos` history).

### HarnessBanner — phase progression
- On `harness_phase_start` SSE → reducer updates `harnessRun.currentPhase` and `harnessRun.phaseName`. Banner re-renders with new fraction + name.
- On `harness_phase_complete` → fraction updates (`n` increments). On `harness_complete` → terminal state (`completed`) shown for 3000ms with `CheckCircle2` green icon (NOT auto-fade unless terminal — same convention as `AgentStatusChip`).
- On reload mid-run: `useChatState` rehydrates `harnessRun` from `GET /threads/{id}/harness/active` → banner appears with current state.

### FileUploadButton — error toast hierarchy
- Client-side fail (size, MIME) → toast immediately, no network call
- Server-side magic-byte fail (4xx) → backend error code maps to existing `t('upload.wrongMime')` (no separate magic-bytes copy — security detail not exposed to user)
- Network/5xx → `t('upload.serverError')`
- Each toast includes the filename when available; max 5000ms auto-dismiss; tap to dismiss

### Reject-while-active behavior (D-02)
- User submits chat message while `harnessRun?.status` IN active set
- Frontend SHOULD NOT pre-block — the existing send button stays enabled (reflects backend authority)
- Backend returns structured `{"error": "harness_in_progress", ...}` → reducer dispatches a banner-style toast using `t('harness.reject.toast')` with the active harness's `harness_type`, `current_phase`, `phase_count` injected
- The user's draft text remains in the input (do NOT clear it) — they can retry after the run completes

### Optional discretionary detail (D-04 follow-up)
- Post-harness summary message (`messages.harness_mode='contract-review'`) renders normally in `MessageView`. **No special badge required** in Phase 20.
- If a small badge is desired in the future (e.g. an inline `Contract Review` chip on the assistant message), reuse the existing `AgentBadge.tsx` pattern with a new variant. Out of scope for Phase 20 verifier; planner may defer entirely.

## State Machine — `harnessRun` slice (frontend)

Lives in `useChatState`. Phase 20 frontend state is read-only relative to the backend; reducer mutates only on SSE events or REST responses.

```
harnessRun:
  | null                                              // no active run, no terminal-state-recently-completed
  | {
      id: uuid                                         // harness_runs.id
      harnessType: string                              // 'contract-review' | 'smoke-echo' | future
      status: 'pending' | 'running' | 'paused'         // active set — locked panel + banner visible
            | 'completed' | 'cancelled' | 'failed'    // terminal — banner shows for 3000ms then null
      currentPhase: number                             // 0-indexed
      phaseCount: number                               // total phases in this harness
      phaseName: string                                // current phase display name
      errorDetail: string | null                       // populated on 'failed'
    }
```

**Transitions** (events that mutate the slice):
- `harness_phase_start` → set `status='running'`, `currentPhase`, `phaseName`
- `harness_phase_complete` → bump `currentPhase`
- `harness_phase_error` → if `code='cancelled'` → set `status='cancelled'`; else → set `status='failed'`, `errorDetail=detail`
- `harness_complete` → set `status='completed'`
- 3000ms after any terminal status → set to `null`
- On thread switch → reset to `null`, refetch via `GET /threads/{id}/harness/active`
