    ```python
         AND no active/terminal harness_runs exists for the thread (D-05).
      2. Streams plain text from a single LLM call (per-harness system prompt baked

      3. Buffers the full stream and matches r"\\s*\\[TRIGGER_HARNESS\\]\\s*$" at end-of-stream.
      4. On match → strips sentinel from final flush, persists user_msg + assistant_reply
    ```

    **Multi-turn history loader (D-08):**
    ```python
    async def load_gatekeeper_history(*, thread_id: str, harness_name: str, token: str) -> list[dict]:
        """Load prior gatekeeper turns: messages WHERE harness_mode=<harness_name> AND no harness_run exists yet.

        D-08: gatekeeper turns persisted with harness_mode=<harness_name>; reload-safe.
        Returns OpenAI-shaped list of {role, content}.
        """
        client = get_supabase_authed_client(token)
        result = (
            client.table("messages")
            .select("role, content, created_at")
            .eq("thread_id", thread_id)
            .eq("harness_mode", harness_name)
            .order("created_at", desc=False)
            .execute()
        )
        rows = result.data or []
        return [{"role": r["role"], "content": r["content"]} for r in rows]
    ```

    **Algorithm (implement precisely):**
    1. Persist `user_message` row with `harness_mode=harness.name` (capture id).
    2. Load prior gatekeeper history via `load_gatekeeper_history`.
    3. Build messages: `[{role: 'system', content: build_system_prompt(harness)}, *history, {role: 'user', content: user_message}]`.
    4. **SEC-04 egress pre-check:** `egress_filter(json.dumps(messages, ensure_ascii=False), registry, None)`. If `tripped`:
       - Persist generic refusal as assistant turn (e.g. "I cannot process this request — sensitive data detected. Please remove it and retry.").
       - Audit-log via `audit_service.log_action(action="gatekeeper_egress_blocked", ...)`.
       - Yield `{"type": "delta", "content": "<refusal text>"}` then `{"type": "gatekeeper_complete", "triggered": False, "phase_count": len(harness.phases), ...}` and return.
    5. Open streaming completion via `openrouter_service.client.chat.completions.create(messages=messages, model=settings.gatekeeper_model_or_default, stream=True)`. Buffer all chunks into `buf: list[str]`. Yield each chunk as `{"type": "delta", "content": chunk_text}` AS YOU RECEIVE IT (low latency).
    6. On stream complete: assemble `full = "".join(buf)`. Apply `SENTINEL_RE.search(full)`:
       - **If match:** strip via `clean = SENTINEL_RE.sub("", full)`. Use the sliding-window guard (below) so the sentinel never streams to the client. Persist `clean` as assistant message with `harness_mode=harness.name`. Call `harness_runs_service.start_run(...)` to create the harness_runs row in `pending` status. Yield:
         ```python
         {"type": "gatekeeper_complete",
          "triggered": True,
          "user_message_id": user_msg_id,
          "assistant_message_id": asst_msg_id,
          "harness_run_id": run_id,
          "phase_count": len(harness.phases)}  # W8 fix
         ```
       - **If no match:** persist `full` as assistant message. Yield gatekeeper_complete with `triggered=False, harness_run_id=None, phase_count=len(harness.phases)`.

    **Sliding-window sentinel guard (D-07 — sentinel never reaches client):** maintain a buffer that holds back the last `len(SENTINEL) + 8` chars (12 + 8 = 20 chars sliding window for trailing whitespace tolerance). Only flush the prefix of the buffer that exceeds the window size as `delta` events. On stream end, check the held-back tail for `SENTINEL_RE` — if present, strip + persist clean version; if absent, flush the tail as the final delta.

    ```python
    @router.post("/threads/{thread_id}/harness/cancel")
    async def cancel_harness(thread_id: str, user: dict = Depends(get_current_user)):
        """Cancel the active harness run for this thread (D-03 + B3 dual-layer cancellation).

        How cancellation propagates to a running engine (HARN-07):
          - This endpoint flips `harness_runs.status='cancelled'` in the DB.
          - The engine (Plan 20-03 Task 4) checks two channels BEFORE each phase:
              (1) An in-process asyncio.Event passed by reference from the SAME
                  request's _gatekeeper_stream_wrapper. This handles the rare case
                  of a same-process Cancel from a code path holding the event ref.
              (2) A DB poll via harness_runs_service.get_run_by_id — checks the
                  .status field. THIS is the channel triggered by the Cancel
                  button in the frontend, which fires this endpoint on a SEPARATE
                  HTTP request from the SSE stream that owns the engine generator.
          - Combined: the user clicks Cancel → this endpoint flips DB row →
            engine notices at the next phase boundary (typically <60s) → engine
            yields harness_phase_error{reason='cancelled_by_user'} → engine exits.
          - Workspace artifacts produced by completed phases are preserved
            (the engine does not roll back phase_results on cancel).
        """
        if not settings.harness_enabled:
            raise HTTPException(status_code=404, detail="harness_disabled")
        active = await harness_runs_service.get_active_run(thread_id=thread_id, token=user["token"])
        if active is None:
            raise HTTPException(status_code=404, detail={"error": "no_active_run"})
        await harness_runs_service.cancel(
            run_id=active["id"],
            user_id=user["id"],
            user_email=user["email"],
            token=user["token"],
        )
        return {"ok": True, "harness_run_id": active["id"], "status": "cancelled"}
    ```
