<objective>
Close the three confirmed bugs from Phase 20 code review that caused must-haves #33, #34, #35 to FAIL in VERIFICATION.md.

**CR-01 (GATE-02):** `gatekeeper.py` — `_WINDOW_SIZE` set to 25 (len(SENTINEL)=17 + 8) does not hold enough buffer when the LLM appends 9 or more trailing spaces after `[TRIGGER_HARNESS]`. The opening `[` is released to the client before end-of-stream detection fires, leaking the sentinel. Fix: increase to `len(SENTINEL) + 8 + 8 = 33` and correct the wrong comment (`12 + 8 = 20` → `17 + 8 + 8 = 33`).

**CR-02:** `harness_engine.py` — `ws = WorkspaceService(token=token)` is assigned inside a `try` block (line 185). If the constructor raises, `ws` is never bound. Lines 338-346 and 384-392 call `_append_progress(workspace=ws)` unconditionally, producing `NameError`. Fix: add `ws: WorkspaceService | None = None` before the try block. `_append_progress` already accepts `workspace=None` as a fallback.

**CR-03:** `backend/app/routers/chat.py` — The 409 `harness_in_progress` JSONResponse (lines 379-387) omits `phase_count`. The frontend's `useChatState.ts` reads `err.body.phase_count` with fallback `harnessRun?.phaseCount ?? 1`. When `harnessRun` is null (fresh page load), the denominator defaults to 1, producing an invalid `phase N/1` fraction. Fix: add `"phase_count": len(h.phases) if h else 0` to the JSONResponse content dict. `h` is already fetched at line 374.

After each fix, add a regression test to the matching test file.
