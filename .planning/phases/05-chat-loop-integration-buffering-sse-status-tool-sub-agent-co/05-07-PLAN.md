# Plan 05-07: Canonical-Only Egress Filtering — Fix D-48 Variant Cascade

## Objective

Stop the egress filter from tripping on D-48 sub-surrogate variants. Variants exist for de-anonymization fragment recognition (D-72 Pass 2 fuzzy match), not for security gating. When spaCy produces false-positive PERSON detections on legal compound nouns ("Governing Law", "Confidentiality Clause"), D-48 stores common English words ("Governing", "Law", "Confidentiality", "Clause") as standalone variant rows. These common words appear naturally in subsequent anonymized text outside of replaced entity spans, tripping the egress filter and permanently blocking the thread after Turn 1.

This is the BLOCKER that makes multi-turn chat unusable in production whenever the LLM response contains legal vocabulary (UAT confirmed in `05-UAT.md` test 5; root cause confirmed via Supabase query on thread `bf1b7325-86e3-4030-b495-8b8013d938b7`).

The fix is architectural, not heuristic: **the egress filter uses canonical real values only**. Variants stay in the registry (de-anonymization needs them); they just stop being egress candidates. The privacy invariant is preserved — every CANONICAL real value is still scanned for, so NER misses on real PII still get blocked.

## Background

### What we observed

### Why the egress trips

The Turn 2 messages payload (after batch anonymization of history) contains the assistant message with literal text like:
```
"6. **Uda Hardiansyah, S.IP**: If necessary, include terms regarding confidentiality."
```

The phrase `"Confidentiality Clause"` was a detected entity span and got replaced with the surrogate. But the standalone word `"confidentiality"` later in the same sentence is NOT an entity span (spaCy does not redetect it as PERSON in lowercase mid-sentence position) — it remains as plain English.

The egress filter scans `registry.entries()` for matches, which includes the variant `("Confidentiality" → ...)`. Word-boundary regex matches `"confidentiality"` in the payload → trip → `EgressBlockedAbort` → empty response.

### Why this is architecturally wrong

D-48 variants exist to support D-72 Pass 2 fuzzy de-anonymization: when the LLM mangles a name slightly ("Suryadi" → "Mr Suryadi" or just "Suryadi"), Pass 2 walks the variant set and rewrites the fragment to the canonical surrogate before final emission. That use case requires variants in the registry.

The egress filter has a different job: prevent CANONICAL real PII from leaking to the cloud LLM. The canonical is `"Ahmad Suryadi"` — that is what NER might miss, and that is what must never leak. The variant `"Ahmad"` alone leaking is not a privacy breach — it's a common Indonesian name. The variant `"Confidentiality"` alone leaking is not a privacy breach — it's a common English noun.

The two paths have been conflated since Phase 3. This plan separates them.

## Tasks

### Task 1: Add `ConversationRegistry.canonicals()` method

**File:** `backend/app/services/redaction/registry.py`

Add a new public method right after `entries()` (around line 149):

```python
def canonicals(self) -> list[EntityMapping]:
    """Return one EntityMapping per unique surrogate_value — the canonical only.

    The D-48 invariant states that all variants in a cluster share the canonical's
    surrogate, and the canonical is always the longest real_value (variants are
    derived BY SUBTRACTION from the canonical: first-only, last-only,
    honorific-prefixed, nickname). Therefore the longest real_value per
    surrogate IS the canonical for that surrogate.

    Used by `egress_filter` (Phase 5 D-94 gap-closure) to prevent variant-only
    matches from tripping the egress when a NER false positive on a compound
    noun (e.g., 'Confidentiality Clause') has produced D-48 variants of common
    English words ('Confidentiality', 'Clause') that naturally appear in
    subsequent anonymized text.

    Variants remain in `entries()` and the DB row set — D-72 Pass 2 fuzzy
    de-anonymization still walks them at full fidelity.

    O(n) one-pass — n is per-thread (typically < 50). Returns a NEW list so
    callers cannot mutate internal state.
    """
    by_surrogate: dict[str, EntityMapping] = {}
    for ent in self._rows:
        existing = by_surrogate.get(ent.surrogate_value)
        if existing is None or len(ent.real_value) > len(existing.real_value):
            by_surrogate[ent.surrogate_value] = ent
    return list(by_surrogate.values())
```

Verify the import path: `EntityMapping` should already be imported at the top (it's the type used by `_rows`).

### Task 2: Switch `egress_filter` to canonicals only

**File:** `backend/app/services/redaction/egress.py`

Replace the candidate-list construction (around line 86-92):

**Before:**
```python
# Build candidate (entity_type, real_value) list.

    # EntityMapping field name confirmed by grepping registry.py before write.
```

**After:**
```python
# Build candidate (entity_type, real_value) list.
# D-48 gap-closure (Plan 05-07): use canonicals() instead of entries() so that
# sub-surrogate variants (first-only / last-only / honorific-prefixed) do NOT
# enter the egress candidate set. Variants stay in entries() for D-72 Pass 2
# fuzzy de-anonymization; only canonical real values gate the cloud-LLM call.
# Privacy invariant preserved: every distinct surrogate has its canonical
# real value scanned for. NER misses on the canonical still trip.
candidates: list[tuple[str, str]] = []
for ent in registry.canonicals():
    candidates.append((ent.entity_type, ent.real_value))
if provisional:
    for real_value in provisional:
        candidates.append(("PERSON", real_value))  # provisional set is PERSON-only
```

Update the module docstring (line 1-14) to clarify scope:

```python
"""Pre-flight egress filter for cloud LLM calls (D-53..D-56, PROVIDER-04, NFR-2).

cloud-LLM call passes through this function with its outbound payload BEFORE
any byte leaves the process. It scans for any case-insensitive word-boundary

  - registry.canonicals() — CANONICAL real values from prior turns of this
```
