<objective>
Wire the frontend SSE consumer side for D-88's two `redaction_status` events (anonymizing / deanonymizing / blocked) so the streaming chat bubble renders a subtle status spinner during the buffer window. This plan is fully INDEPENDENT of the backend plans 05-01..05-04 — it can execute in parallel with Plan 05-04 in Wave 3 and ship without any backend changes (the new SSEEvent variant is a no-op when the backend doesn't emit it). Three small files: a TypeScript type extension, a React hook dispatch addition, and three i18n strings.

Purpose: BUFFER-02 (FR-6.2) requires the user-visible progress indicator. Without it, the buffered single-batch delivery (D-87) makes the streaming experience feel broken — user sees a long pause, then a wall of text, with no indication anything is happening. The two `redaction_status` events bracket the buffer window and the spinner provides continuous feedback. SC#5 backward-compat: when redaction is OFF, the existing CHAT-06 SSE handling MUST be byte-identical — no regression in delta / tool_* / agent_* dispatch behavior.

Output: Three files modified. ~30-60 LOC total change. The actual visual rendering of the spinner (in `ChatMessage.tsx` or equivalent) is INTENTIONALLY out of scope for this plan — Plan 05-05 ships the state + i18n strings; the visual UI polish is a follow-up that the i18n/UI maintainer owns. The hook return value exposes `redactionStage` so any consumer that wants to render a spinner has the data.
</objective>
