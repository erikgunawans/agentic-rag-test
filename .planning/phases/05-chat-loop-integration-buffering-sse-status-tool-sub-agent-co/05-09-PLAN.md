# Phase 05 Plan 09: Frontend pii_redaction_enabled Toggle (Gap Closure)

## Objective

Close the UAT gap surfaced during 05-UAT re-verification: Plan 05-08 added the DB column, API contract, and service-layer wiring for `pii_redaction_enabled`, but the frontend `AdminSettingsPage.tsx` was never updated to render the toggle. Admins had no way to switch PII redaction on/off from the browser without direct API calls.

## Tasks

### Task 1: Add `pii_redaction_enabled` to SystemSettings TypeScript interface

In `frontend/src/pages/AdminSettingsPage.tsx`, extend the `SystemSettings` interface with the new optional boolean field, mirroring the backend `SystemSettingsUpdate` Pydantic model from Plan 05-08.

### Task 2: Render the master PII redaction toggle in the PII section

Add a checkbox toggle at the top of the `activeSection === 'pii'` block, before the status badges, since it is the master on/off switch. Use the existing controlled-checkbox pattern (e.g., `pii_missed_scan_enabled`):
- `checked={form.pii_redaction_enabled ?? true}` (defaults to true to match DB default)
- `onChange={(e) => updateField('pii_redaction_enabled', e.target.checked)}`
- Followed by a `<Separator />` to visually group it as the master switch

### Task 3: Add bilingual i18n strings

Add `admin.pii.redactionEnabled.label` and `admin.pii.redactionEnabled.desc` to both the Indonesian (default) and English translation maps in `frontend/src/i18n/translations.ts`. The description warns: "When off, messages reach the LLM without anonymization."

## Verification

- TypeScript: `cd frontend && npx tsc --noEmit` exits 0
- Lint: no new errors in the two modified files
- Production: open `/admin/settings`, navigate to PII section → toggle is visible, reflects DB value (default true), can be flipped, save persists to DB (subject to 60s `get_system_settings()` cache TTL)
