Exact content:

```dockerfile
# SandboxDockerfile — Phase 10 / D-P10-03
# Build:  docker build -f SandboxDockerfile -t lexcore-sandbox:latest .
# Push:   docker tag lexcore-sandbox:latest <dockerhub-user>/lexcore-sandbox:latest
#         docker push <dockerhub-user>/lexcore-sandbox:latest
#
# llm-sandbox pulls this image at session-creation time. The backend service
# (backend/Dockerfile) is unrelated — that runs the FastAPI app.
#
# Pre-installed packages (D-P10-03): pandas, matplotlib, python-pptx, jinja2,
# requests, beautifulsoup4, numpy, openpyxl, scipy, ipython.
#
# All RUN commands MUST run BEFORE the USER switch — same lesson as
# backend/Dockerfile spaCy install (CLAUDE.md gotchas: EACCES on .cache/pip
# when running as non-root after USER switch).

FROM python:3.11-slim

WORKDIR /sandbox

# System deps for matplotlib (libpng, freetype) and openpyxl/python-pptx (libxml2/libxslt)
RUN apt-get update \
 && apt-get install -y --no-install-recommends \
        libpng-dev libfreetype6-dev libxml2 libxslt1.1 \
 && rm -rf /var/lib/apt/lists/*

# D-P10-03: pre-install scientific + document-generation packages.
# No version pins — use whatever pip resolves at build time. Re-pin if
# reproducibility becomes critical.
RUN pip install --no-cache-dir \
        pandas \
        matplotlib \
        python-pptx \
        jinja2 \
        requests \
        beautifulsoup4 \
        numpy \
        openpyxl \
        scipy \
        ipython

# Output dir — generated files land here (D-P10 / SANDBOX-04). World-writable
# because the runtime user (1000:1000) needs to create files.
RUN mkdir -p /sandbox/output && chmod 777 /sandbox/output

# Non-root execution (security posture parity with backend/Dockerfile USER switch)
USER 1000:1000

# No CMD — llm-sandbox manages container exec. The image just provides the
# Python runtime + libraries; llm-sandbox `docker exec`s into it.
```

**Why no CMD:** llm-sandbox creates the container with its own keep-alive command (typically `tail -f /dev/null` or similar) and execs Python code via `docker exec` per call. Setting a CMD would be ignored or interfere with llm-sandbox's lifecycle.

**Build/push is manual:** This task does NOT build or push the image. The image must be built and pushed to Docker Hub (or any registry the Railway DOCKER_HOST can pull from) before `sandbox_enabled=true` is set in production. Document this as a deploy-time step in the plan summary.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Sandbox container → host filesystem | Docker isolation; only the bind-mounted /sandbox/output is writable from inside the container |
| Sandbox container → host Docker socket | NONE — sandbox containers are leaf nodes; they do NOT have docker.sock mounted in. Only the backend service mounts the socket to spawn sandboxes. |
| Configuration secrets → settings load | sandbox_enabled is a boolean, not a secret; sandbox_docker_host is a path, not a credential |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-07 | Elevation of Privilege | Sandbox container running as root could escape to host | mitigate | `USER 1000:1000` non-root switch in SandboxDockerfile (Task 2). Verified by acceptance criterion `grep -q "^USER 1000:1000"` |
| T-10-08 | Tampering | Outdated/compromised Docker image pulled from Hub | accept | Image is published under our Docker Hub account; pull is by tag `:latest`. For v1.1, accept latest-tag risk. Future: pin to digest `sha256:...` for supply-chain hardening |
| T-10-09 | Information Disclosure | Sandbox image contains secrets baked into layers | mitigate | No `ARG` / `ENV` for credentials in SandboxDockerfile. No `COPY` of source code. Image is purely runtime + pip packages |
| T-10-10 | Spoofing | Settings load with invalid sandbox_image string accepted | accept | No image-name validator. If pull fails at runtime, llm-sandbox surfaces the error and the tool returns `{"error": ...}` to the LLM. Acceptable failure mode |
| T-10-11 | Denial of Service | sandbox_enabled=true on host without Docker daemon | accept | Runtime failure surfaces as tool error per Plan 03; no startup gate. Operator responsibility — Railway env config doc must call this out |
</threat_model>
