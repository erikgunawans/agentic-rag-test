<objective>
Thread the sandbox `stream_callback` through `_run_tool_loop` in `chat.py` so
that during an `execute_code` tool invocation, stdout/stderr lines flow as SSE
`code_stdout`/`code_stderr` events to the frontend in real time. Resolve two
planner alerts:

1. **Async-generator + callback impedance (alert #1)** — `_run_tool_loop` is
   itself an async generator (yields `(event_type, data)` tuples). A callback
   cannot directly `yield` from inside the generator's frame. **Solution: an
   `asyncio.Queue` adapter** — the callback `.put()`s events; the loop drains
   the queue between tool_start and tool_result and yields each event using the
   existing `(event_type, data)` contract. The downstream branches A and B at
   chat.py lines 459 and 525 already understand "any other event_type" → SSE
   passthrough, so adding two new event types preserves backward compatibility.

2. **PII redaction interaction with code stdout (alert #2)** — When
   `redaction_on=True`, Phase 5 D-89 mandates skeleton-only emit for tool
   output. **Decision: anonymize each line before emit** (option (b) per the
   planner alert). This honors the D-89 privacy invariant — no real PII leaks
   into the SSE stream — while still showing the user meaningful (surrogate)
   output. We use the existing `anonymize_tool_output` helper from
   `app.services.redaction` to keep the pipeline consistent with how
   tool_start/tool_result handle redaction.

Purpose: SANDBOX-03 — line-by-line `code_stdout`/`code_stderr` SSE events
during execution. Without this plan, sandbox output only appears in the
final `tool_result` blob — defeating the streaming UX promised by SANDBOX-03.

Output: Modified `chat.py`. NO changes to event_generator's outer flow, NO
changes to redaction status events, NO changes to existing tool_start /
tool_result skeleton-emit logic.
</objective>

The new logic — surgical insertion:

```python
                # Phase 10 D-P10-05/06: queue adapter for execute_code streaming.
                # Planner alert #1 resolution: _run_tool_loop is itself an async
                # generator, so the callback CANNOT yield directly. Instead, the
                # callback enqueues; we drain the queue between tool_start and
                # tool_result while awaiting the execute_tool task.
                sandbox_event_queue: "asyncio.Queue | None" = None
                sandbox_callback = None
                if func_name == "execute_code":
                    sandbox_event_queue = asyncio.Queue()

                    async def sandbox_stream_callback(event_type: str, line: str):
                        """D-P10-05/06: enqueue code_stdout/code_stderr lines.

                        Planner alert #2 resolution: when redaction_on=True, anonymize
                        the line before enqueue so SSE never emits real PII (D-89
                        invariant). This is a per-line anonymization — different from
                        tool_start/tool_result skeleton-emit because SANDBOX-03 requires
                        meaningful streaming output, and skeleton-only would defeat the UX.
                        """
                        emit_line = line
                        if redaction_on and registry is not None:
                            try:
                                anon = await anonymize_tool_output(
                                    {"line": line}, registry, redaction_service,
                                )
                                # anonymize_tool_output returns same shape; extract line
                                if isinstance(anon, dict):
                                    emit_line = anon.get("line", line)
                            except Exception as _exc:
                                logger.warning(
                                    "sandbox stream anon failed err=%s — emitting skeleton",
                                    _exc,
                                )
                                emit_line = ""  # skeleton fallback (D-89 safer default)
                        await sandbox_event_queue.put({
                            "type": event_type,
                            "line": emit_line,
                            "tool_call_id": tc["id"],
                        })
                    sandbox_callback = sandbox_stream_callback
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Sandbox stdout/stderr line → SSE stream → frontend | Untrusted code output crosses to client. PII redaction filters apply when redaction_on=True |
| `tc["id"]` (LLM-emitted tool_call_id) → SSE payload | LLM-controlled string echoed in payload. Used for client-side correlation only |
| asyncio.Queue → backpressure | Unbounded queue could grow if sandbox produces output faster than SSE can flush |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-26 | Information Disclosure | Real PII in sandbox stdout leaks to SSE when redaction_on=True | mitigate | Per-line `anonymize_tool_output` call inside sandbox_stream_callback (alert #2 resolution). Skeleton-emit fallback (`emit_line = ""`) on anonymization failure |
| T-10-27 | Tampering | LLM-controlled tc["id"] could be a script tag or other injection vector when echoed in SSE | mitigate | tool_call_id is JSON-serialized via `json.dumps` (chat.py existing pattern); the frontend MUST treat it as opaque text. JSON encoding handles HTML/JS injection at this layer |
| T-10-28 | Denial of Service | Unbounded queue grows if sandbox spams output (e.g., infinite print loop) | accept | Per-call execution timeout (D-P10-12, 30s default) bounds total output volume. Queue is process-local and ephemeral; OOM extremely unlikely within 30s. Future: add queue maxsize if observed |
| T-10-29 | Repudiation | Sandbox events emitted but not persisted to messages.tool_calls | accept | Phase 11 (MEM-01) handles persistence. For Phase 10, SSE is fire-and-forget; full stdout/stderr is persisted in code_executions row by Plan 04 |
| T-10-30 | Information Disclosure | Existing tool_loop_buffer holds anonymized lines briefly before flush | accept | Lines are already anonymized; buffering for ~30s during execute_code call is acceptable. Special-case bypass (Edit B branches A/B) means sandbox events skip the buffer entirely |
</threat_model>
