Phase 10 uses 4-segment paths per D-P10-13: `{user_id}/{thread_id}/{execution_id}/{filename}`.
- user_id = segment [1] (same index as skills)
- thread_id = segment [2]
- execution_id = segment [3]

```sql
CREATE TABLE IF NOT EXISTS public.code_executions (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  thread_id     UUID NOT NULL REFERENCES public.threads(id) ON DELETE CASCADE,
  code          TEXT NOT NULL,
  description   TEXT,
  stdout        TEXT NOT NULL DEFAULT '',
  stderr        TEXT NOT NULL DEFAULT '',
  exit_code     INTEGER NOT NULL DEFAULT 0,
  execution_ms  INTEGER NOT NULL DEFAULT 0,
  status        TEXT NOT NULL CHECK (status IN ('success','error','timeout')),
  files         JSONB NOT NULL DEFAULT '[]'::jsonb,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

**2. Indexes** (two — for thread-scoped list endpoint and user-scoped admin queries):

```sql
CREATE INDEX IF NOT EXISTS idx_code_executions_thread
  ON public.code_executions(thread_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_code_executions_user
  ON public.code_executions(user_id, created_at DESC);
```

**3. RLS** — per D-P10-15: SELECT for own + super_admin; INSERT for own; NO UPDATE, NO DELETE (immutable audit log):

```sql
ALTER TABLE public.code_executions ENABLE ROW LEVEL SECURITY;

CREATE POLICY "code_executions_select"
  ON public.code_executions
  FOR SELECT
  USING (
    user_id = auth.uid()
    OR (auth.jwt() -> 'app_metadata' ->> 'role') = 'super_admin'
  );

CREATE POLICY "code_executions_insert"
  ON public.code_executions
  FOR INSERT
  WITH CHECK (user_id = auth.uid());

-- No UPDATE / DELETE policies — execution log is immutable audit record (D-P10-15).
```

**4. Bucket** (D-P10-13, mirror 035 lines 105–107):

```sql
INSERT INTO storage.buckets (id, name, public)
VALUES ('sandbox-outputs', 'sandbox-outputs', false)
ON CONFLICT (id) DO NOTHING;
```

**5. Storage RLS** — 4-segment path, only segment [1] checked. Use `storage.objects.name` (not bare `name`) to avoid the ambiguous-column bug from migration 035 cycle-2 fix. Add an explanatory header comment about the 4-segment path:

```sql
-- ============================================================
-- 5. STORAGE RLS — sandbox-outputs (4-segment path: user_id/thread_id/execution_id/filename)
-- ============================================================
-- D-P10-13: path = {user_id}/{thread_id}/{execution_id}/{filename}.
-- Only segment [1] (user_id) is gated. Unlike skills (3-segment with FK on segment [2]),
-- we have no parent-row constraint per-execution; the bucket is private so SELECT is
-- gated solely by ownership of the top-level folder.
-- BUG FIX FROM 035: Use objects.name (not bare `name`) inside any subquery to avoid
-- PostgreSQL resolving `name` against an outer table.

CREATE POLICY "sandbox-outputs SELECT"
  ON storage.objects
  FOR SELECT
  USING (
    bucket_id = 'sandbox-outputs'
    AND (storage.foldername(objects.name))[1] = auth.uid()::text
  );

-- INSERT: backend uploads via service-role (bypasses RLS) per CONTEXT.md line 102.
-- The policy still gates direct user uploads — caller UID must match segment [1].
CREATE POLICY "sandbox-outputs INSERT"
  ON storage.objects
  FOR INSERT
  WITH CHECK (
    bucket_id = 'sandbox-outputs'
    AND (storage.foldername(objects.name))[1] = auth.uid()::text
  );

-- No DELETE policy — generated files retention-managed at bucket level (future).
-- No UPDATE policy — files are immutable.
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Backend service-role → storage.objects | Service-role bypasses RLS; could write objects to wrong user's folder if path constructed incorrectly |
| User JWT → code_executions table SELECT | RLS enforces `user_id = auth.uid()`; super_admin claim bypass via JWT app_metadata |
| Client-side direct upload (none) | Bucket is private; clients only see signed URLs (1-hour TTL via D-P10-14) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Information Disclosure | code_executions SELECT cross-user | mitigate | RLS policy `code_executions_select` filters `user_id = auth.uid()`; super_admin gated on `app_metadata.role` claim (signed JWT, server-set only) |
| T-10-02 | Tampering | code_executions UPDATE/DELETE post-write | mitigate | Zero UPDATE/DELETE policies — table is INSERT-only audit log per D-P10-15. Verified by acceptance criterion `! grep -qE "FOR (UPDATE|DELETE).*code_executions"` |
| T-10-03 | Information Disclosure | sandbox-outputs cross-user file read | mitigate | Storage SELECT policy gates on `(storage.foldername(name))[1] = auth.uid()::text`; bucket public=false; signed URL 1-hour TTL (D-P10-14) limits exposure |
| T-10-04 | Spoofing | User-controlled segment [2] (thread_id) or [3] (execution_id) | accept | RLS only enforces ownership of segment [1]; user CAN write to arbitrary thread_id/execution_id paths under their own user_id folder. Acceptable — service-role uploader (Plan 03) constructs paths from server-validated context, and even if a user spoofed values, they remain confined to their own RLS-bounded folder |
| T-10-05 | Denial of Service | Unbounded code_executions row growth per user | accept | No row-level quota in v1.1. Indexes ensure read performance stays bounded. Future: add retention policy or per-user execution rate limit if Railway storage costs become a concern |
| T-10-06 | Elevation of Privilege | Migration 036 misapplied / partially applied | mitigate | All DDL uses `IF NOT EXISTS` / `ON CONFLICT DO NOTHING` — re-runnable. `supabase db push` BLOCKING gate (Task 3) ensures atomic application before downstream plans build |
</threat_model>
