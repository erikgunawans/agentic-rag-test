## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| router state to route renderer | The user's clicked nav item triggers a NavLink to React Router transition. Untrusted input does not cross this boundary; `path` is a hard-coded string literal. |
| translations.ts to DOM | Translation values are inserted into JSX as text content via `t()`. React auto-escapes by default, so injected strings are rendered as text, not HTML. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01-01 | Tampering | translations.ts (i18n strings) | accept | Translations are static, version-controlled source code. Any tampering requires a code commit reviewed via PR — the same gate as all other strings in the app. No runtime user-controlled translation injection exists. |
| T-09-01-02 | Information disclosure | nav.skills label | accept | Label is the same string ("Skills") shown to every authenticated user. No PII or tenant-specific data in nav labels. |
| T-09-01-03 | Elevation of privilege | /skills route | mitigate | Route is registered as a child of `<AuthGuard>` only — NOT under `<AdminGuard>`. Any authenticated user can access. Backend RLS (Phase 7) is the authoritative gate; the route is a public navigation surface. Confirmed by reading App.tsx:46-49 — AuthGuard wraps AppLayout outlet. |
| T-09-01-04 | XSS via translation interpolation | t('skills.emptySearch', { query }) where query comes from search input | mitigate | The interpolator at I18nContext.tsx:29-33 uses `value.replace('{key}', v)` and the result is rendered as a JSX text node, which React escapes. No raw-HTML injection sinks are used. Search query content rendered through `t()` is XSS-safe by default. Plan 02 must render translated strings only as JSX text children, never via raw-HTML props. |
</threat_model>
