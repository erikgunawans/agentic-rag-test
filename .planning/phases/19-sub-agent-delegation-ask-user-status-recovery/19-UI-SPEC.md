## Component Inventory

All new components introduced by Phase 19.

### 1. AgentStatusChip (`frontend/src/components/chat/AgentStatusChip.tsx`)

**Purpose:** D-26 — header slot status indicator for the active agent run.

**Placement:** Rendered in `frontend/src/layouts/AppLayout.tsx` inside the `<main>` element's inner chat header row (above the message thread, below the top bar). The chat page (`ChatPage.tsx` or equivalent) must expose a header slot; if none exists, AgentStatusChip is rendered at the top of the chat content column as a sticky bar flush with the top edge.

**States and visual spec:**

| State | Display | Icon | Background | Text |
|-------|---------|------|------------|------|
| `null` (no active run) | Hidden — renders nothing | — | — | — |
| `working` | Visible | Pulsing dot (4px circle, `bg-primary animate-pulse rounded-full`) | `bg-zinc-100 dark:bg-zinc-800` | chip label |
| `waiting_for_user` | Visible | `MessageCircleQuestion` size 14, `text-purple-600 dark:text-purple-400` | `bg-purple-50 dark:bg-purple-950/40` | chip label |
| `complete` | Visible, auto-fades after **3000ms** | `CheckCircle2` size 14, `text-green-600 dark:text-green-400` | `bg-green-50 dark:bg-green-950/40` | chip label |
| `error` | Visible, persists until dismissed or new run | `AlertCircle` size 14, `text-red-600 dark:text-red-400` | `bg-red-50 dark:bg-red-950/40` | chip label |

**Chip anatomy:**
```
[icon 14px] [label text-sm font-semibold] 
px-2 py-1 rounded-full inline-flex items-center gap-1.5
border border-current/20
```

**Auto-fade implementation:** `useEffect` with `setTimeout(3000)` sets `agentStatus` to `null` on `complete`. The chip uses `transition-opacity duration-300` so the fade is 300ms. Total visible time: 3000ms at full opacity + 300ms fade-out.

**Transition between states:** `transition-all duration-200` on the chip container. State change = unmount current chip + mount new chip (React key change on status value). No cross-fade needed; 200ms is imperceptible at chip scale.

**Accessibility:**
- `role="status"` on the chip container.
- `aria-live="polite"` on the chip container. This ensures screen readers announce state changes without interrupting ongoing speech.
- `aria-label` matches the full text label (not just the icon name).
- When chip is hidden (`null` state), render an empty `aria-live` container that remains in the DOM so announcements work reliably: `<div role="status" aria-live="polite" className="sr-only" />`.

**Source:** D-26 (CONTEXT.md) — locked decision.

### 2. TaskPanel (`frontend/src/components/chat/TaskPanel.tsx`)

**Purpose:** D-25 — sidebar panel listing all sub-agent task cards for the active thread.

**Placement:** Sidebar slot, rendered to the right of the thread content (same slot pattern as WorkspacePanel and PlanPanel). Render order from left to right: IconRail → ThreadPanel → main content → PlanPanel → WorkspacePanel → TaskPanel. TaskPanel is the rightmost panel.

**Visibility rule:** Render whenever `tasks` slice has at least one entry for the current thread. Decoupled from Deep Mode toggle. Same rule as WorkspacePanel (`files.length > 0`).

**Panel surface:** `bg-background` (NO `backdrop-blur`, NO glass — persistent panel, CLAUDE.md rule). `border-l border-border/50`. Width: `w-72 shrink-0` — matches PlanPanel and WorkspacePanel.

**Panel header:**
```
px-4 py-3 border-b border-border/50
h3: text-sm font-semibold text-foreground  [t('taskPanel.title')]
  + task count: text-muted-foreground font-normal  ({tasks.size})
collapse button: h-6 w-6 rounded hover:bg-accent transition-colors
  ChevronDown (expanded) / ChevronRight (collapsed), size 14
```

**Task card anatomy (one card per task_id):**

```
rounded-lg border bg-muted/40 p-3 text-sm space-y-2
  [Row 1] status icon (16px) + description (text-sm text-foreground leading-tight)
  [Row 2] context_files chips (if any):
    each chip: bg-zinc-200 dark:bg-zinc-800 px-2 py-1 rounded-full text-[11px] font-normal
    max-w truncation with title tooltip (base-ui tooltip, render prop pattern)
  [Row 3] nested tool calls (if any, indented pl-4):
    each row: font-mono text-[11px] text-muted-foreground + tool_call_id text-[10px] opacity-60
    matches SubAgentPanel.tsx explorer_tool_calls rendering exactly
  [Row 4 — terminal state only] result preview (complete) or error message (error):
    complete: text-xs text-muted-foreground line-clamp-2
    error: text-xs text-red-600 dark:text-red-400 line-clamp-2
```

**Task status icons (size 16, shrink-0):**

| Status | Icon | Class |
|--------|------|-------|
| `running` | `Loader2` | `animate-spin text-purple-500` |
| `complete` | `CheckCircle2` | `text-green-500` |
| `error` | `AlertCircle` | `text-red-500` |

These mirror PlanPanel StatusIcon exactly (same icons, same colors) — executor MUST use identical classes for visual consistency.

**Empty state:** TaskPanel renders `null` when `tasks` slice has zero entries. No empty-state illustration or copy is needed — the panel simply disappears. This matches WorkspacePanel behavior (`if (files.length === 0) return null`).

When all tasks are `complete` or `error` (no `running` tasks), the panel remains visible until the thread is changed or reloaded. This gives the user a persistent record of what ran. It does NOT auto-hide on all-complete.

**Collapse behavior:** Local `useState` toggle (collapsed/expanded). Matches PlanPanel and WorkspacePanel. Collapsed state shows only the panel header bar.

**Scroll:** `flex-1 overflow-y-auto px-4 py-3` inside the content area. Cards stacked with `space-y-2`.

**Accessibility:**
- `role="complementary"` on the `<aside>` element.
- `aria-label={t('taskPanel.title')}` on the `<aside>`.
- Each task card: `aria-label` composed as `[status] [description substring]` (e.g., "Running: Analyze clause risk").
- Spinner: `aria-hidden="true"` (motion icon, not meaningful to screen readers; status announced via card aria-label).

**Source:** D-25 (CONTEXT.md) — locked decision. Placement (new TaskPanel.tsx, not SubAgentPanel extension) confirmed here — SubAgentPanel is a history-only inline renderer for the v1.0 multi-agent classifier, structurally incompatible with the new panel slot pattern.

### 3. Question-bubble variant in MessageView (`frontend/src/components/chat/MessageView.tsx`)

**Purpose:** D-27 — special rendering for an assistant message whose `tool_calls` contains an `ask_user` invocation without a matching `tool_result`.

**Detection:** `isAskUserQuestion` flag derived in MessageView from message's `tool_calls` JSONB: present when a call with `name === 'ask_user'` exists and no `tool_result` row with matching `tool_call_id` exists in `toolResults`. This flag is computed per-message.

**Visual diff from normal assistant bubble:**

| Property | Normal assistant bubble | Question bubble |
|----------|------------------------|-----------------|
| Left border | none | `border-l-[3px] border-primary` |
| Icon | none | `MessageCircleQuestion` size 16, `text-primary`, positioned left of text content |
| Background | default (bg-card or bg-muted) | same — no background change |
| Text color | `text-foreground` | `text-foreground` — no change |
| Question text source | streaming/content | extracted from `ask_user` tool call `question` argument |
| Spacing above question text | standard | `pl-4` to indent text after the left-border stripe |

**Question text extraction:** The displayed text is the `question` parameter from the `ask_user` tool call arguments JSON, NOT the raw assistant content. If the assistant message has additional text content before the tool call, render both: content first (normal), then the question-bubble block below.

**Structural HTML pattern:**
```tsx
<div className="flex items-start gap-2 border-l-[3px] border-primary pl-3">
  <MessageCircleQuestion size={16} className="text-primary shrink-0 mt-0.5" />
  <p className="text-sm text-foreground leading-relaxed">{question}</p>
</div>
```

**Accessibility:**
- The question-bubble container: `role="note"` to signal it is supplemental content requiring user attention.
- `aria-label={t('askUser.questionBubble.ariaLabel')}` on the container.
- No focus management change — the question renders in the natural document flow. The user sees it as the last message and types a reply in the normal input box.

**Source:** D-14, D-27 (CONTEXT.md) — locked decisions.

## Interaction Contract

### AgentStatusChip lifecycle

```
SSE agent_status { status: 'working' }
  → agentStatus = 'working'
  → chip mounts, pulsing dot visible

SSE agent_status { status: 'waiting_for_user' }
  → agentStatus = 'waiting_for_user'
  → chip transitions (200ms), MessageCircleQuestion icon, purple tint
  → question-bubble rendered as last assistant message (separate mechanism via MessageView)

SSE agent_status { status: 'complete' }
  → agentStatus = 'complete'
  → chip transitions (200ms), green check
  → setTimeout(3000): opacity-0 transition (300ms) → agentStatus = null → chip unmounts

SSE agent_status { status: 'error' }
  → agentStatus = 'error'
  → chip transitions (200ms), red alert
  → persists until thread is changed or new run starts
  → user clicks input box and re-sends message to retry (no special retry button)
```

### TaskPanel card lifecycle

```
SSE task_start { task_id, description, context_files }
  → tasks[task_id] created with status='running'
  → card mounts with Loader2 spinner

SSE tool_start { task_id, ... }
  → tasks[task_id].toolCalls pushed
  → nested tool row appears in card

SSE tool_result { task_id, ... }
  → tasks[task_id].toolCalls last entry updated with result
  → tool row updated (no visual change at row level; only status chip logic)

SSE task_complete { task_id, result }
  → tasks[task_id].status = 'complete', result stored
  → spinner → CheckCircle2 (instant, no transition animation)
  → result preview rendered (line-clamp-2)

SSE task_error { task_id, error }
  → tasks[task_id].status = 'error', error stored
  → spinner → AlertCircle
  → error text rendered (line-clamp-2, red)
```

### Question-bubble interaction

```
SSE ask_user event received
  → agentStatus = 'waiting_for_user' (handled by AgentStatusChip)
  → final assistant message in thread has tool_calls containing ask_user invocation
  → MessageView.isAskUserQuestion = true
  → question-bubble variant renders with border-left accent + MessageCircleQuestion icon
  → input box remains normal (no special "reply mode")
  → user types reply and clicks Send (existing flow)
  → server resume-detection branch routes reply as ask_user tool result
  → new SSE stream opens, agentStatus transitions to 'working'
  → question-bubble remains in thread history (append-only)
```

## Implementation Notes for Executor

These are clarifications to prevent ambiguity during plan 19-07 execution.

1. **AgentStatusChip render location:** AppLayout.tsx currently has no chat-header slot. The `<main>` element renders `<Outlet>`. The chat page (`ChatPage.tsx` or the route component) must render AgentStatusChip at the top of the chat column, BEFORE the message list scroll container. It must be `sticky top-0 z-10` so it stays visible while scrolling through long threads. Do NOT add it inside AppLayout.tsx itself — AppLayout does not have access to chat-specific state.

2. **TaskPanel render location:** ChatPage (or the route component equivalent) renders the panel row layout. The current layout renders PlanPanel and WorkspacePanel. TaskPanel is added to the right of WorkspacePanel in the same flex row. No changes to AppLayout.tsx required for TaskPanel.

3. **SubAgentPanel.tsx is NOT modified.** It remains the history-reload renderer for v1.0 multi-agent classifier tool calls. TaskPanel is a new independent component. They coexist without conflict.

4. **question-bubble does not affect message persistence.** The `isAskUserQuestion` flag is derived at render time from the message's `tool_calls` JSONB content. No new DB columns or message fields are needed for the question-bubble visual.

5. **AgentStatusChip auto-fade timer:** The timer is set inside the component via `useEffect` when `agentStatus === 'complete'`. If the user switches threads before the 3s timer fires, the component unmounts and the timer is cleaned up in the effect cleanup function (`clearTimeout`). No memory leak.

6. **context_files chip truncation:** Chip text is the file path. Max width: `max-w-[120px] truncate`. The base-ui tooltip (using `render` prop, NOT `asChild`, per CLAUDE.md tooltip shim) shows the full path on hover.

7. **`agentStatus` null guard:** All three components (AgentStatusChip, TaskPanel via `tasks` slice check, question-bubble via `isAskUserQuestion` flag) must handle null/empty state by rendering `null` (React empty). Do not render placeholder skeleton states for these components.
