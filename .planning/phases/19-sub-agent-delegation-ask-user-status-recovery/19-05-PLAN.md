<objective>
Wire `ask_user` end-to-end. Two cooperating pieces:

1. **Pause path** — `ask_user` tool registered + `chat.py` ask_user dispatch handler that persists `agent_runs.status='waiting_for_user'` (via 19-02 service), emits the canonical `agent_status` (Site B per D-16) + `ask_user` SSE events, yields `done`, and returns from the generator (D-01). **This dispatch handler is the single canonical owner of the `agent_status='waiting_for_user'` emission.** 19-06 documents this ownership but does NOT add a second emission.

2. **Resume path** — `stream_chat` entry-point branch that detects `agent_runs.status='waiting_for_user'` for the thread, transitions status to `working`, and re-enters `run_deep_mode_loop` with `body.message` injected as the `ask_user` tool's result string (D-04, D-15).

Purpose: Without this, `ask_user` is a no-op; without the resume branch, the thread is permanently stuck in `waiting_for_user`. This plan completes the close-and-resume protocol that distinguishes Phase 19 from a one-shot agent.

Output: `_ASK_USER_SCHEMA` + `_ask_user_executor` appended below 19-04's `_register_sub_agent_tools` block; ask_user dispatch handler inside `run_deep_mode_loop` (with the canonical Site B `agent_status` yield); resume-detection branch at top of `stream_chat`; tool tests + integration test for the two-request resume sequence.

Note: D-09 retention of `ask_user` in the sub-agent's tool list is asserted in 19-03 (the filter owner). This plan does NOT duplicate that assertion.
</objective>
