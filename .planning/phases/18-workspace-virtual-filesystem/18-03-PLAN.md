        name="write_file",

                "name": "write_file",

                        "content": {"type": "string", "description": "Full file contents."},

                    "required": ["file_path", "content"],

        schema={
            "type": "function",
            "function": {
                "name": "read_file",
                "description": "Read a workspace file by relative path.",

                    "properties": {

                },

        loading="immediate",

**Step C — Define the four executor functions** in `tool_service.py`. Each is `async`, accepts `**kwargs` (the LLM-supplied tool args plus framework-supplied context like `thread_id` and `token`), and returns a dict.

The framework already passes `thread_id` and the user's auth token to executors via the dispatch wrapper — find the existing pattern (search for an executor in `tool_service.py` that uses `thread_id` parameter; e.g., the sandbox `execute_code` executor).

```python
async def _workspace_write_file_executor(
    *, thread_id: str, token: str, file_path: str, content: str, **_unused
) -> dict:
    ws = WorkspaceService(token=token)
    return await ws.write_text_file(thread_id, file_path, content, source="agent")

async def _workspace_read_file_executor(
    *, thread_id: str, token: str, file_path: str, **_unused
) -> dict:
    ws = WorkspaceService(token=token)
    return await ws.read_file(thread_id, file_path)

async def _workspace_edit_file_executor(
    *, thread_id: str, token: str, file_path: str, old_string: str, new_string: str, **_unused
) -> dict:
    ws = WorkspaceService(token=token)
    return await ws.edit_file(thread_id, file_path, old_string, new_string)

async def _workspace_list_files_executor(
    *, thread_id: str, token: str, **_unused
) -> dict:
    ws = WorkspaceService(token=token)
    files = await ws.list_files(thread_id)
    return {"ok": True, "files": files, "count": len(files)}
```

## STRIDE Threat Register
| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-11 | T (Tampering) | LLM passes a path with `..` | mitigate | service-layer validator (plan 18-02) returns structured error before any DB hit |
| T-18-12 | I (Information disclosure) | LLM uses thread_id from another user | mitigate | RLS via authed client; the chat-loop wires the AUTHED user's thread_id, not LLM-supplied |
| T-18-13 | E (Elevation) | feature flag bypass | mitigate | `if WORKSPACE_ENABLED:` is the registration gate; with flag false, executor is also unreachable since registry is empty |
| T-18-14 | T | LLM sets `source="upload"` to fake user upload | accept | the executor hardcodes `source="agent"` for write_file; LLM cannot influence the source field |
</threat_model>
