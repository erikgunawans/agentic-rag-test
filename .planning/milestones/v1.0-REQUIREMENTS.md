## v1.0 Requirements — PII Redaction System (54 requirements)

**All 54 requirements shipped. PERF-02 has a partial hardware-dependent validation item.**

### PII Detection (PII-*)

- ✓ **PII-01** — v1.0 Phase 1: 16 PII entity types via Presidio NER (PERSON, EMAIL_ADDRESS, PHONE_NUMBER, LOCATION, DATE_TIME, URL, IP_ADDRESS, CREDIT_CARD, US_SSN, US_ITIN, US_BANK_NUMBER, IBAN_CODE, CRYPTO, US_PASSPORT, US_DRIVER_LICENSE, MEDICAL_LICENSE)
- ✓ **PII-02** — v1.0 Phase 1: Two-pass NER detection — 0.7 threshold (surrogate), 0.3 threshold (hard-redact)
- ✓ **PII-03** — v1.0 Phase 1: Entity-type bucket config from `PII_SURROGATE_ENTITIES` and `PII_REDACT_ENTITIES` env vars
- ✓ **PII-04** — v1.0 Phase 1: UUID-segment false-positive filter to prevent document-ID corruption
- ✓ **PII-05** — v1.0 Phase 1: Presidio NER engine, gender-detection model, and nickname dictionary loaded as lazy singletons at startup

### Anonymization (ANON-*)

- ✓ **ANON-01** — v1.0 Phase 1: Faker-generated realistic surrogate values (PERSON → realistic name, EMAIL → realistic email, PHONE → realistic phone, etc.)
- ✓ **ANON-02** — v1.0 Phase 1: Hard-redact entities replaced with `[ENTITY_TYPE]` placeholder
- ✓ **ANON-03** — v1.0 Phase 1: Surrogate collision-free — no reuse of real surname or first name from same input batch
- ✓ **ANON-04** — v1.0 Phase 1: Gender-matched surrogates for PERSON entities (female → female, ambiguous → random)
- ✓ **ANON-05** — v1.0 Phase 1: Programmatic find-and-replace preserving position and whitespace
- ✓ **ANON-06** — v1.0 Phase 1: Indonesian-specific gender lookup table (78 first names: 40M/32F/6 ambiguous)

### Conversation-Scoped Entity Registry (REG-*)

- ✓ **REG-01** — v1.0 Phase 2: Per-thread persisted registry (Supabase `entity_registry` table, migration 029)
- ✓ **REG-02** — v1.0 Phase 2: Case-insensitive lookups (same entity in different casings → same surrogate)
- ✓ **REG-03** — v1.0 Phase 2: Consistency guarantee — same real entity always maps to same surrogate within a thread
- ✓ **REG-04** — v1.0 Phase 2: Registry survives thread reload (persisted to DB, reloaded on resume)
- ✓ **REG-05** — v1.0 Phase 2: Hard-redacted placeholders never stored in registry (intentionally one-way)

### Entity Resolution (RESOLVE-*)

- ✓ **RESOLVE-01** — v1.0 Phase 3: Three resolution modes — `algorithmic` (Union-Find coreference), `llm` (LLM-assisted), `none`
- ✓ **RESOLVE-02** — v1.0 Phase 3: Algorithmic mode uses Union-Find clustering with 3 merge rules (nickname, partial-name, title-stripped)
- ✓ **RESOLVE-03** — v1.0 Phase 3: LLM mode operates only on pre-anonymized surrogates (never real values for cloud provider)
- ✓ **RESOLVE-04** — v1.0 Phase 3: Non-PERSON entities (emails, phones, URLs) use exact-match normalization, never sent to resolution LLM

### De-Anonymization (DEANON-*)

- ✓ **DEANON-01** — v1.0 Phase 2: Surrogate → real value replacement for user display
- ✓ **DEANON-02** — v1.0 Phase 2: Case-insensitive matching (surrogate variants → original real value)
- ✓ **DEANON-03** — v1.0 Phase 4: Fuzzy matching — `algorithmic` (Jaro-Winkler ≥ 0.85), `llm`, or `none` modes
- ✓ **DEANON-04** — v1.0 Phase 4: Placeholder-tokenized 3-phase pipeline (surrogate→placeholder → fuzzy-match → placeholder→real)
- ✓ **DEANON-05** — v1.0 Phase 4: Hard-redact `[ENTITY_TYPE]` placeholders survive de-anonymization unchanged in all modes

### Response Buffering & SSE (BUFFER-*)

- ✓ **BUFFER-01** — v1.0 Phase 5: Full LLM response buffering when redaction is active (no streaming partial surrogates to user)
- ✓ **BUFFER-02** — v1.0 Phase 5: `redaction_status` SSE events with `stage: anonymizing` and `stage: deanonymizing` for UX
- ✓ **BUFFER-03** — v1.0 Phase 5: Sub-agent reasoning events suppressed during buffered redaction

### System-Prompt Guidance (PROMPT-*)

- ✓ **PROMPT-01** — v1.0 Phase 4: Main-agent and all sub-agent system prompts instruct LLM to preserve exact surrogate formatting (no abbreviation, no reformatting)

### Missed-PII Secondary Scan (SCAN-*)

- ✓ **SCAN-01** — v1.0 Phase 4: Optional LLM-driven secondary scan over already-anonymized text (gated by `PII_MISSED_SCAN_ENABLED`)
- ✓ **SCAN-02** — v1.0 Phase 4: Missed-scan runs across all three entity-resolution modes
- ✓ **SCAN-03** — v1.0 Phase 4: Provider follows global `LLM_PROVIDER` setting (or per-feature override)
- ✓ **SCAN-04** — v1.0 Phase 4: Scan results validated against hard-redact entity set (invalid types discarded)
- ✓ **SCAN-05** — v1.0 Phase 4: Primary NER engine re-runs after missed-scan replacement (single-re-run cap via D-76)

### LLM Provider Configuration (PROVIDER-*)

- ✓ **PROVIDER-01** — v1.0 Phase 3: Global `LLM_PROVIDER` (`local` | `cloud`) for all auxiliary features
- ✓ **PROVIDER-02** — v1.0 Phase 3: Local-provider mode uses `LOCAL_LLM_BASE_URL` + `LOCAL_LLM_MODEL`; operates on raw real content (no third-party egress)
- ✓ **PROVIDER-03** — v1.0 Phase 3: Cloud-provider mode uses `CLOUD_LLM_BASE_URL` / `CLOUD_LLM_MODEL` / `CLOUD_LLM_API_KEY`; auxiliary calls operate on pre-anonymized data only
- ✓ **PROVIDER-04** — v1.0 Phase 3: Pre-flight egress filter scans payload against conversation registry; aborts on any real-value match
- ✓ **PROVIDER-05** — v1.0 Phase 3: Both providers use OpenAI-compatible APIs (`/v1/chat/completions`); local works with LM Studio / Ollama
- ✓ **PROVIDER-06** — v1.0 Phase 3: Configuration available through env vars and admin settings API/UI
- ✓ **PROVIDER-07** — v1.0 Phase 3: Per-feature provider overrides (`ENTITY_RESOLUTION_LLM_PROVIDER`, `MISSED_SCAN_LLM_PROVIDER`, `TITLE_GEN_LLM_PROVIDER`, `METADATA_LLM_PROVIDER`, `FUZZY_DEANON_LLM_PROVIDER`) falling back to global `LLM_PROVIDER`

### Tool-Call & Sub-Agent Coverage (TOOL-*)

- ✓ **TOOL-01** — v1.0 Phase 5: `search_documents` tool — LLM-generated query (with surrogates) de-anonymized before search; results re-anonymized before returning to LLM
- ✓ **TOOL-02** — v1.0 Phase 5: `query_database` tool — query de-anonymized before execution; results re-anonymized before returning to LLM
- ✓ **TOOL-03** — v1.0 Phase 5: `web_search` tool — search pattern de-anonymized; results re-anonymized in next chat-loop iteration
- ✓ **TOOL-04** — v1.0 Phase 5: Sub-agents (document analyzer, KB explorer, nested sub-agent) share parent's redaction-service instance; symmetric anonymize-input / de-anonymize-output through nested invocations

### Embedding Provider Configuration (EMBED-*)

- ✓ **EMBED-01** — v1.0 Phase 6: `EMBEDDING_PROVIDER=local|cloud`; default `cloud` preserves existing OpenAI-embeddings flow (RAG-02)
- ✓ **EMBED-02** — v1.0 Phase 6: Local embedding mode supports OpenAI-API-compatible local endpoint (e.g., bge-m3/nomic-embed-text via Ollama); switching does NOT trigger automatic re-embedding (deployer-managed migration)

### Observability & Tracing (OBS-*)

- ✓ **OBS-01** — v1.0 Phase 1: All redaction operations traced via pluggable observability provider (`TRACING_PROVIDER=langsmith|langfuse|""`)
- ✓ **OBS-02** — v1.0 Phase 6: Debug-level logs capture entities detected, surrogates assigned, fuzzy matches, missed-PII scan results, UUID-filter drops, LLM provider per call, egress-filter results — all with `thread_id` correlation field
- ✓ **OBS-03** — v1.0 Phase 6: Each LLM call logs its resolved provider (after per-feature override resolution) for audit

### Reliability & Performance (PERF-*)

- ✓ **PERF-01** — v1.0 Phase 1: Presidio NER engine, gender-detection model, and nickname dictionary loaded once at startup as lazy singletons
- ✓* **PERF-02** — v1.0 Phase 6: Anonymization completes in under 500ms for typical chat messages (< 2000 tokens). *Partial: 2000ms hard CI gate passed; 500ms primary target unconfirmed on dev hardware (1939ms on macOS Python 3.14). Run `pytest tests/services/redaction/test_perf_latency.py -m slow` on CI/Railway to confirm.*
- ✓ **PERF-03** — v1.0 Phase 2: Concurrent requests writing to same registry serialized via asyncio lock (race verified against live Supabase UNIQUE constraint)
- ✓ **PERF-04** — v1.0 Phase 6: Graceful degradation when LLM provider unavailable — entity resolution → algorithmic, missed-PII scan → skipped, title/metadata → 6-word template; failures logged, never crash loop, never leak PII
