# Requirements: LexCore v1.1 — Agent Skills & Code Execution

**Status:** ✅ SHIPPED 2026-05-02 (v0.5.0.0)
**Total v1.1 requirements:** 26 — all completed.

## Milestone Goal

Transform LexCore's chat interface into a customizable AI agent platform where users create and manage reusable skills, attach resource files to those skills, execute sandboxed Python code, and maintain persistent tool memory across conversation turns.

---

## v1.1 Requirements (final state)

### Agent Skills Core (SKILL-*)

- [x] **SKILL-01**: User can create a skill with name, description, and instructions via a form dialog → Phase 7 (`POST /skills` + Phase 9 editor)
- [x] **SKILL-02**: User can trigger AI-guided skill creation that uses the built-in `skill-creator` skill conversationally via chat → Phase 8 (`load_skill` tool) + Phase 9 ("Buat dengan AI" button)
- [x] **SKILL-03**: User can browse all visible skills (own + global) in a searchable list with global/disabled badges → Phase 9 (search panel + ownership badges)
- [x] **SKILL-04**: User can edit skill properties (name, description, instructions) and toggle enabled/disabled → Phase 7 (`PATCH /skills/{id}`) + Phase 9 (editor form)
- [x] **SKILL-05**: User can delete their own private skills → Phase 7 (`DELETE /skills/{id}`) + Phase 9 (delete with name-confirm dialog)
- [x] **SKILL-06**: User can toggle a skill between private and global (share/unshare with all users) → Phase 7 (`PATCH /skills/{id}/share`) + Phase 9 (share button + GLOBAL badge)
- [x] **SKILL-07**: System injects enabled skills as a lightweight name/description catalog into the LLM system prompt → Phase 8 (`build_skill_catalog_block()`, fail-soft, capped at 20)
- [x] **SKILL-08**: LLM can call `load_skill` tool to fetch full instructions when user query matches a skill description → Phase 8 (`tool_service.py`)
- [x] **SKILL-09**: LLM can call `save_skill` tool to persist a new skill after AI-guided creation flow → Phase 8 (`tool_service.py`)
- [x] **SKILL-10**: A global `skill-creator` seed skill is created during setup/migration → Phase 7 Migration 034 (UUID `00000000-0000-0000-0000-000000000007`)
- [x] **SKILL-11**: A "Skills" tab appears in top navigation alongside Chat and Documents → Phase 9 (Zap icon in IconRail, `aria-label="Skill"`, `/skills` route)

### Skill Building-Block Files (SFILE-*)

- [x] **SFILE-01**: User can upload files to a skill from the skill editor (own skills only) → Phase 8 (`POST /skills/{id}/files` + 10 MB middleware) + Phase 9 (upload UI)
- [x] **SFILE-02**: `load_skill` tool response includes a table of attached files (name, size, type) → Phase 8
- [x] **SFILE-03**: LLM can call `read_skill_file` tool to fetch content of a specific skill-attached file → Phase 8
- [x] **SFILE-04**: User can preview text file content in a slide-in panel with copy and download buttons → Phase 9 (480px drawer, monospace pre, binary download card, Esc/backdrop/× close)
- [x] **SFILE-05**: User can delete files attached to their own skills → Phase 8 (`DELETE /skills/{id}/files/{file_id}`) + Phase 9

### Code Execution Sandbox (SANDBOX-*)

- [x] **SANDBOX-01**: LLM can execute Python code in a sandboxed Docker container via `execute_code` tool → Phase 10 (Plan 10-04)
- [x] **SANDBOX-02**: Python session persists variables across `run()` calls within a thread (30-min TTL, auto-cleanup) → Phase 10 (Plan 10-03 — IPython session-per-thread)
- [x] **SANDBOX-03**: stdout/stderr stream to frontend via SSE events (`code_stdout`, `code_stderr`) in real-time → Phase 10 (Plan 10-05)
- [x] **SANDBOX-04**: Files generated in `/sandbox/output/` are uploaded to Supabase Storage and returned as signed URLs → Phase 10 (Plan 10-03 + Plan 10-06)
- [x] **SANDBOX-05**: `execute_code` tool is only registered when `SANDBOX_ENABLED=true` → Phase 10 (Plan 10-04 hard gate)
- [x] **SANDBOX-06**: All executions logged in `code_executions` table (code, stdout, stderr, exit code, timing, status) → Phase 10 Migration 036
- [x] **SANDBOX-07**: Chat shows inline Code Execution Panel with code preview, streaming output, and file downloads → Phase 11 (`CodeExecutionPanel.tsx`, 359 lines)
- [x] **SANDBOX-08**: Custom Docker image pre-installs common packages (python-pptx, pandas, matplotlib, jinja2, requests, beautifulsoup4, etc.) → Phase 10 (`SandboxDockerfile`, `lexcore-sandbox:latest`)

### Skills Open Standard (EXPORT-*)

- [x] **EXPORT-01**: User can export a skill as a `.zip` file in agentskills.io-compatible format (SKILL.md frontmatter + instructions + categorized subdirs: scripts/, references/, assets/) → Phase 7 (`GET /skills/{id}/export`, fixed in `faa5403`)
- [x] **EXPORT-02**: User can import skills from a `.zip` file (max 50MB), supporting single-skill and bulk-import ZIP formats → Phase 7 (`POST /skills/import` + `SkillsUploadSizeMiddleware`)
- [x] **EXPORT-03**: Import validates name/description and reports per-skill errors without blocking other skills in a bulk import → Phase 7 + Phase 9 (summary modal with created/error counts)

### Persistent Tool Memory (MEM-*)

- [x] **MEM-01**: After each tool execution, full result string is stored in `messages.tool_calls` JSONB alongside existing name/arguments/status/summary fields → Phase 11 (Plan 11-01 ToolCallRecord schema + Plan 11-04 4 ctor sites incl. silent-bug fix on multi-agent success path)
- [x] **MEM-02**: Conversation history load reconstructs tool call messages in LLM-expected format (assistant tool-call → tool result → assistant text) → Phase 11 (Plan 11-04 `_expand_history_row`)
- [x] **MEM-03**: LLM can reference data from earlier tool calls (UUIDs, search results, file listings) in follow-up questions without re-executing tools → Phase 11 UAT step 9 confirmed end-to-end (no `code_stdout`/`code_stderr` SSE events on follow-up turn)

## Out of Scope (v1.1) — confirmed at close

These were explicitly OUT of v1.1 scope and remain so:
- Scheduled skill execution (auditability concerns for legal platform)
- Skill access control beyond private/global (per-team/role ACL deferred)
- Sandbox multi-language support (Python only)
- Real-time collaborative skill editing
