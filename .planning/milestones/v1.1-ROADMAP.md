## Overview

Transform LexCore's chat interface into a customizable AI agent platform where users create and manage reusable skills, attach resource files to those skills, execute sandboxed Python code with live streaming output, and maintain persistent tool memory across conversation turns. Closes the gap between "RAG-augmented chat" and "agent platform with code execution".

**Headline capabilities shipped:**
- Skills system end-to-end (DB + RLS + storage bucket + 8 REST endpoints + LLM tool integration + frontend with editor/file management/share/import-export)
- Code Execution Sandbox (Docker isolated Python with IPython session persistence, 30-min TTL, llm-sandbox wrapper, real-time SSE streaming, signed-URL file download)
- Code Execution UI (CodeExecutionPanel React component with code preview / status pill / live timer / dark terminal / file cards) wired into ToolCallList router
- Persistent tool memory (`messages.tool_calls.calls[]` carries full tool_call_id + status + output, reconstructed on history load so the LLM references prior results across turns without re-execution)

**Privacy invariant preserved:** All v1.0 PII redaction guarantees still hold. The new `execute_code` tool's stdout/stderr stream is anonymized through the same registry-based filter before reaching the cloud LLM, and the new `messages.tool_calls.calls[].output` field carries already-anonymized text.

## Milestone Summary

### Decisions anchored

- **Composite skill ownership** — `user_id` (current owner) + `created_by` (creator). Sharing transfers `user_id=NULL`; `created_by` is preserved for attribution.
- **Skills as a single-row LLM-facing catalog block** — fail-soft, capped at 20 entries, RLS-scoped at build time. Avoids exploding the system prompt as the skill set grows.
- **`SANDBOX_ENABLED` as a hard gate** — `execute_code` tool is absent from `get_available_tools()` when false. Prevents accidental sandbox invocation in environments without Docker daemon. Operational cost: requires explicit env flip in Railway, easy to forget on first deploy of v1.1.
- **`tool_call_id` + `status` are optional+nullable** — pre-Phase-11 records still typecheck; no migration. Phase 11's silent-bug fix (multi-agent success path now appends ToolCallRecord) closes the only correctness gap.
- **50KB head-truncation as Pydantic field_validator** — single source of truth for the cap. All 4 ToolCallRecord ctor sites get truncation for free.
- **`sandboxStreams` Map keyed by tool_call_id (UUID)** — reset at 3 lifecycle sites (thread switch, send, post-stream finally). UUID keys make stale entries safe even with unconditional pass-through to `ToolCallList`.
- **Popover `asChild` shim** — added to mirror the `tooltip.tsx` pattern. Caught by Vercel's `tsc -b` (project-references build mode) but missed by local `tsc --noEmit`. Worth promoting `tsc -b` to pre-flight in `/deploy-lexcore`.

### Issues encountered

- **Phase 07 export `relative_path` KeyError** — surfaced post-ship in UAT; fixed in `faa5403` with regression tests.
- **Phase 10 migration applied via `supabase db query -f` workaround** — duplicate `024_*.sql` files prevented standard `db push`. Documented in 10-01-SUMMARY.
